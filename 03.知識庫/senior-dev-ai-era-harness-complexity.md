## 核心論點

AI 越強，**管複雜度的能力反而越值錢**。被淘汰的不是工程師，是「把價值定義在打字快、API 背得多」的那種工作方式。

---

## 深藍時刻類比

1997 年深藍打敗西洋棋世界冠軍 → 很多人以為棋手要失業了。
結果：棋手棋力不降反升，長出「人機協作」新下法。
被淘汰的是「只會背棋譜」的下法，不是棋手本身。

**我們這行正在經歷自己的深藍時刻。**

---

## 溫徹斯特神秘屋比喻

Winchester Mystery House：38 年沒有藍圖、每天早上隨手畫草圖施工的結果：
- 樓梯爬上去撞天花板
- 門打開是一面牆
- 門開在二樓外牆，推開直接摔下去
- 每一塊工都做得細緻，合在一起沒人看得懂

**AI 在專案裡幹的就是這件事：**
- 它只負責當下這個 prompt，腦袋裡沒有整棟房子的概念
- 看起來能跑、CI 綠燈，單獨沒問題
- 三個月後：同一件事有四個地方在做、有 code 永遠跑不到、有 API 一呼叫就崩

> 真正可怕的不是哪一行寫錯，而是整棟房子大到沒有任何人能在腦袋裡裝得下它。

## 資深工程師新的核心價值

| 舊價值（被 AI 取代中） | 新價值（AI 越強越稀缺） |
|----------------------|----------------------|
| 打字快、語法熟 | 管複雜度 |
| API 背得多 | 模組切割、邊界設計 |
| 寫 code | **讀 code、審 code** |
| 調好提示詞 | 設計 Harness（治具） |

---

## Harness（治具）概念

> 與其鑽研怎麼把提示詞寫漂亮，不如替 AI 蓋一個治具。

提示詞再好，輸出還是飄。能規模化的做法：
1. 給 AI **清楚的規格**
2. **一組能自動跑的測試**
3. **明確的及格線**

讓 AI 在這個框架裡自己撞牆、自己修正，不是每次靠人盯對話框。

**把品質標準「工程化」，讓系統自動把關。**

→ 這就是為什麼 Claude Code 的 CLAUDE.md + type check + lint 如此重要

---

## BDD 重新變重要

當 AI 負責「怎麼做」，人最該守住的是「要做出什麼行為」。

- 用具體例子把需求講清楚 = 給人看的溝通，也是驗收 AI 的測試
- 規格越清楚，AI 越有辦法自動驗證自己
- SBE（Specification by Example）的價值在 AI 時代更高

---

## 讀 code > 寫 code

以前瓶頸在「寫」，現在 AI 一口氣生一大坨，瓶頸換到：
- **看得懂**
- **判斷對不對**
- **抓得出哪裡有雷**

會讀、會審、會問對問題的人，才壓得住 AI 的產出。

---

## 一句話結論

> 「背鍋俠」翻成正經話：**最後那個要對結果負責的人。這個位子，AI 還坐不上去。**
