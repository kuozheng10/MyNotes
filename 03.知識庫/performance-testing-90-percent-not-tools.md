## 核心洞見

工具是最後 10% 的事。前面 90% 的功課，沒有捷徑。

工具再強，前面五件事沒做好，跑出來的數字跟「占卜」一樣沒意義。

## 五件關鍵事

**一、過關標準要明確（SLA/SLO）**

沒有門檻數字就不是工程，是占卜。

例：訂單 API P95 < 800ms、500 RPS 下錯誤率 < 0.1%、CPU < 70%、記憶體無持續成長。

**二、工作負載模型要像真實使用者**

壞的做法：login → 狂打某 API → logout，1000 thread 一起衝。

正確做法需要：
- 真實使用者旅程比例（80% 瀏覽、15% 加購、5% 結帳）
- 真實思考時間
- 真實資料分布（熱門商品 vs 平均分布，結果天差地遠）
- 真實尖峰模式

**三、可觀測性比測試工具更重要**

P95 = 3 秒，然後呢？沒有 APM、DB slow query log、CPU/memory/IO 時序圖 → 只知道「很慢」，不知道下一步。

花三週學 k6，出問題時連 DB connection pool 用了幾條都不知道。

**四、環境一致性**

- 8 核測完推到 2 核正式環境：沒有參考價值
- 1 萬筆資料測的查詢，跟 1000 萬筆是兩個世界

**五、看百分位數，不是平均值**

平均 200ms 聽起來很棒，P99 = 8 秒 → 每 100 人就有 1 人體驗很糟。

永遠看 **P50 / P95 / P99** 和分布圖，平均值會騙人。

## 三個診斷問題

工程師說要做效能測試，問他：

1. 過關標準是什麼？（要具體數字）
2. 模擬的使用者行為長怎樣？根據什麼資料？
3. 測試失敗，你會看哪些指標找瓶頸？

答得出來 → 給任何工具都能做出有價值的測試。
答不出來 → 給最貴的工具也只產出漂亮但無用的報表。

## 為什麼大家執著問工具

- **工具具體，思維抽象**：「學 k6」可以打勾、有影片有認證；「建合理負載模型」沒辦法兩小時學完
- **偽工作的感覺**：寫腳本、跑測試、產報表看起來很忙，但沒定義目標就是工程版偽工作
- **履歷導向**：「精通 JMeter」好寫，「能跟 PM 談出合理 P99」難寫，但後者價值高 10 倍
- **迴避暴露**：問工具比問「我們的效能目標是什麼」安全，後者會暴露沒想清楚

## AI Coding 時代的影響

寫測試腳本 AI 可以幫你，但「定義什麼是好的效能」還是只有人能做。

AI 大幅降低實作成本 → 工具差距被縮小 → 思維差距被放大。
