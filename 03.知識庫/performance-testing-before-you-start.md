# 效能測試開始前的五件事（先想清楚比工具更重要）

> 來源：agile3uncles.com
> 整理日期：2026-06-01

---

## 核心問題

> 工具選好了、腳本也寫了，壓下去跑出一堆漂亮的數字——然後沒人知道這到底算過還是不過。

**問題不在工具，在「還沒搞清楚要測什麼，就先開始壓」。**

---

## 開始前的五件事

### 1. 為什麼要做這次測試？

效能測試不是例行公事，是在回答**具體問題**：
- 驗證新功能能不能撐住尖峰？
- 懷疑某次改版讓 API 變慢了？
- 想知道系統天花板在哪？

問題不同 → 測試設計完全不同。目標不清楚，連「該設計哪種負載」都無法決定。

---

### 2. 什麼叫「過」？（最常被跳過、最致命）

沒有明確準則，測完只能說「感覺還行」。

**最少需要三個門檻：**
- 回應時間（**必須指定 P95/P99，不是平均值**）
- 吞吐量（TPS/RPS）
- 可接受的錯誤率

> 平均值在效能測試裡很會騙人。

**用業務語言訂：**
- ✅ 「結帳在尖峰時段 P95 < 2秒、錯誤率 < 0.1%」
- ❌ 「工程師自己拍腦袋一個數字」

---

### 3. 負載模型要貼近真實

「同時上線 1000 人」≠「同一瞬間 1000 個 request」

**腳本裡要正確設定：**
- 尖峰人數
- 使用者操作路徑
- Think time（動作間停頓時間）—設錯會嚴重高估/低估系統能力
- 流量型態（平穩 or 爆衝）

拿不到真實數據 → 跟 PO、營運把假設講明白。

---

### 4. 環境和資料要夠真實

| 地雷 | 影響 |
|------|------|
| 測試機規格是正式環境 1/10 | 數字沒參考價值 |
| 資料量太小 | Cache 永遠命中，壓出虛假結果 |
| Cache 永遠熱 | 低估冷啟動時的真實壓力 |

**動作**：盤點環境規格、資料筆數、cache 冷熱狀態，標註與正式環境的落差。

---

### 5. 要「看得見」系統內部

只有前端回應時間 → 知道「慢」，但不知道「為什麼慢」。

**一開始就要接上：**
- APM（Application Performance Monitoring）
- Server 端資源：CPU / 記憶體 / DB 連線數

> 效能測試的真正價值是「定位瓶頸」，不是給一個數字。

## AI Coding 時代的新風險

### 風險 1：AI 產的程式碼「功能對、效能爛」

- Prompt 通常只寫功能需求，不寫非功能需求
- AI 傾向給「能跑」的程式碼，不是「能擴展」的
- 常見陷阱：N+1 查詢、缺索引、該非同步卻同步呼叫
- 單元測試全綠、demo 很順 → **一上量就垮**

→ **效能基準要建得比以前更早**

### 風險 2：AI 寫的壓測腳本不能照單全收

- AI 能快速生腳本，但不知道你真實的負載模型
- 很可能混淆 concurrent（並發）vs simultaneous（同時）
- **腳本看起來很專業，模型卻是錯的**

→ 產得越快，越要有人驗證「壓的是不是你想壓的」

### 風險 3：開發變快，效能債也累積更快

- 以前：一週改 3 個地方
- 現在：一天改 30 個
- 等上線前才做大壓測 → 太晚

→ **把效能準則前移，變成 CI 自動把關的關卡**

---

## 一句話結論

> 工具和腳本，AI 都能幫你加速。  
> 「要測什麼、什麼叫過、真實負載長什麼樣」——這三件事，到今天還是得由懂業務、懂系統的人想清楚。  
> 而且因為 AI 讓效能陷阱藏得更深，這三件事反而比以前更需要在一開始就講明白。
