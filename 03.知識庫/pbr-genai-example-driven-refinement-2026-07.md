## 這是什麼

一篇虛構情境故事（Zebra 團隊），示範怎麼用 GenAI 把一句話的模糊需求，在一場 PBR（Product Backlog Refinement）會議裡快速轉化成 agent 讀得懂、人也驗證得了的具體規格。核心工具是「先讓 AI 生出具體範例，範例引爆團隊討論，討論結果落成 spec + 可執行驗收測試」。

## 故事情境

**背景教訓**：Zebra 團隊三個月前導入 coding agent。上個 sprint 有人把一句話需求（「根據用戶消費紀錄，提供不同權益」）直接丟給 agent，agent 沒有反問，自己選了一組「合理」預設值——用累積消費金額算等級、退貨完全不扣。做出來的東西能動但邏輯全錯，team review 兩天，最後整包重寫。

**這次的做法**：這張卡片一樣只有一句話「做 VIP 客戶分級」。Scrum Master **大仁** 先喊停：「agent 不會問『你到底要什麼』，它只會補上它猜的答案。需求釐清的成本不會消失，只是搬到 review 的時候爆炸。」PO **小宜** 承認自己也還沒想清楚，只知道「要讓常買的客戶感覺被重視」。

大仁建議：把想法丟進 GeneAI，先挖出具體範例。小宜輸入一句話後，GeneAI 十秒內回三個具體情境：

1. 王小姐：三個月內消費五次、每次約2000元 → VIP 2級，享專屬折扣
2. 李先生：一次消費10,000元，但間隔超過半年 → 只列 VIP 1級
3. 陳小姐：消費紀錄裡有退貨，淨消費未達標 → 無法升等

這三個範例像火花一樣點燃討論：QA **小茵** 問「退貨算不算消費？爭議訂單怎麼辦？」，**阿哲** 問「VIP等級要顯示在哪裡？」GeneAI 接著追問四個引導性問題：升等即時算還是批次算？要不要有降級規則？跟現有會員身份會不會重疊？哪幾條要寫成 agent 可自動執行的驗收測試？

大仁的關鍵一句：「這些問題，今天不回答，agent 明天寫 code 時還是會遇到。差別只在於，今天回答的是我們，明天回答的是它的預設值。」

## 結構化討論 → spec 產出

團隊把一句話需求拆成**五份 spec 檔**放進 repo，每份包含：
- **背景**：為什麼需要這個功能（提升客戶忠誠度）
- **範例**：王小姐/李先生/陳小姐這種具體情境直接寫成表格，不用再翻譯一次
- **技術標註**：需要哪些資料表/API/前端元件，哪些決定還沒定案、agent 不准自己猜（例如 VIP 圖示視覺呈現要跟設計師另外討論，標成待確認）
- **驗收條件**：寫成可執行測試場景，agent 生完 code 要先跑這幾條

過程中的技術細節：後端工程師**小任**指出如果用「每週批次升等」，agent 要跨資料表抓訂單和退貨紀錄，這個限制得寫進 spec，不然agent八成會直接在API裡即時算。小茵列出三個異常情境（計算週期內多次退貨、爭議訂單要不要先剔除、退貨金額超過消費金額導致淨額變負時等級怎麼算），直接標成第一批要寫的驗收測試。

## 結局：隔天早上的十分鐘

第二天早上 agent 根據第一份 spec 交出實作。小茵跑異常情境測試，抓到一個 bug：爭議訂單沒被剔除。因為 spec 裡有陳小姐這個明確範例，她十分鐘就確認問題出在哪，agent 半小時內修完，測試全綠。

阿哲的對照：「上次那張一句話需求，光搞清楚『它做的這樣算對嗎』就花了半天。這次十分鐘。」

大仁總結：「Agent 寫 code 很快，這已經不用懷疑。我們真正花時間的地方，是確認它做出來的東西對不對。昨天那一個半小時的 PBR，買回來的就是今天的十分鐘。」

小宜的結論：「原來我的工作沒有被 AI 取代，只是換了一個地方發揮——以前我寫需求給工程師看，現在我跟整個團隊一起，把需求寫到人和 agent 都不會誤解。」

## 對派哥的意義

- 這篇故事示範的做法，跟你自己在 Claude Code 那邊既有的習慣（實作前先列✅正常/⚠️邊界/❌錯誤情境，也就是 Specification by Example, SBE）幾乎是同一套方法論，只是這裡多了「先讓 AI 生範例當引子，範例引爆團隊討論」這一步——對你來說可以反過來用：丟需求給我之前，先自己想1-2個具體情境，或請我先幫你生幾個情境範例來確認理解對不對，再進入實作，跟你「新功能開發SOP: 先寫SBE→Codex實作→Claude review」的既有流程是同一條路數的補強
- 跟同作者昨天那篇 [[kanban-ai-agent-human-in-loop-2026-07]] 是系列作，那篇談「agent做完事後誰審查（In Review閘門）」，這篇談「agent動工前需求怎麼問清楚（PBR階段）」，兩篇合起來剛好是 agent 協作的「事前」跟「事後」兩端把關
- 跟 [[bdd-behavior-driven-development-ai-era]] 核心命題完全一致：「AI會把含糊需求以一百倍速產出一百倍垃圾」，這篇提供了一個具體、可操作的會議流程示範，可以當成之後帶討論/寫spec時的參考腳本
