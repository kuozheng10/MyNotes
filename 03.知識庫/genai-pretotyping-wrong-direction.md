## 核心問題

> 「AI 讓路變平了，但如果你導航設錯，只會讓你更全速地衝向懸崖。」

GenAI 最大的隱藏危機：**做錯方向的事卻做得太快**。

---

## 具體研究數據（2026-04-28 補充）

| 研究 | 數據 |
|------|------|
| DX 公司 12.1萬開發者 | 90%+ 使用 AI 工具，生產力提升中位數仍只有 10% |
| METR 對照實驗 | 開發者感覺快 20%，實際慢 19%（39% 感知落差） |
| Standish Group | 企業應用 64% 功能「很少/從未」被使用，只有 7% 總是被用 |
| Pendo（數百家客戶）| 80% 功能很少或從不被使用 |
| AI 程式碼品質 | 邏輯錯誤 ×1.75、安全漏洞 ×1.57 |

**核心推論**：AI 速度提升 30%，但 64-80% 功能沒人用 → AI 讓你用 1.3 倍速製造更多垃圾。

**Gartner 2026 預測**：90% 軟體工程師將從「寫程式」轉向「AI 流程編排」。

## GenAI 的三個效率錯覺

| 錯覺 | 真相 |
|------|------|
| 秒產程式碼 = 驗證過 | 驗證速度跟不上生成速度，驗證成了瓶頸 |
| 丟給 AI 寫就好 | 略過討論，每人理解的邏輯完全對不上 |
| AI 產出的 UI 很專業 = 正確 | AI 只做機率性聯想，不懂你的業務邏輯 |

**慘痛案例**：3 天用 AI 做完看板系統，客戶說「我要的是跨部門資源衝突預警」。
→ 產出速度 ×3，打掉重練的挫折感也 ×3。

## 動手 Coding 前先做這四件事

### 1. Pre-totyping（虛擬原型）
用最廉價的方式（紙筆、Miro 方塊）做「假功能」，直接問客戶：「這幾個按鈕能解決問題嗎？」
→ 花 1 小時驗證，省掉 3 天打掉重練

### 2. Customer Journey Map（用戶旅程圖）
觀察客戶從接任務到結案的全過程 → 真正痛點往往不是介面，而是「資料格式不統一」

### 3. Impact Mapping（影響力地圖）
問「為什麼做？對誰有幫助？」→ 發現表面需求背後的真實動機

### 4. Specification by Example（實例化需求）⭐
寫出具體真實情境範例再餵給 AI → 產出精準度有質的飛躍
```
❌ 「當資源重疊時發送通知」
✅ 「當 A 部門的張三在同一週被分配到專案 X 和 Y 共 48 小時，
   系統在週一早上 9 點主動推播給兩位 PM」
```

## AI 時代開發者的新價值

| 舊思維 | 新思維 |
|--------|--------|
| 先寫再修 | 先驗證再寫 |
| 寫長篇文件 | 產出實例（TDD/SBE） |
| 滿足需求 | 理解動機 |
| 對 Code 負責 | 對解決方案負責 |

> **GenAI 是噴射引擎，不是自動導航。**

## 對派哥的直接應用

派哥的 CLAUDE.md 已有 SBE 規則：
> 「說『實作 X』前先列 ✅正常/⚠️邊界/❌錯誤 情境」

這篇文章是同一概念的完整版。My Wallet Trip、cc_processor 新功能都應該先做：
1. Pre-totyping（紙上流程 or 假 UI）
2. 列出 3-5 個具體使用情境
3. 再叫 Claude 實作
