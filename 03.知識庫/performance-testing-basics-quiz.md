# 效能測試基本觀念

> 整理日期：2026-05-30

---

## 核心概念速查

| 概念 | 定義 |
|------|------|
| 效能測試定位 | **非功能性測試**，驗證回應時間、吞吐量、資源使用率 |
| 核心觀察指標 | 回應時間 + 吞吐量 + 資源使用率，**三者必須一起看** |
| Load Testing | 驗證系統在**預期正常與尖峰負載**下能否達到效能目標 |
| Stress Testing | 刻意**超過**系統能力，觀察崩潰點與恢復能力 |
| Soak Testing | 在**中等負載下長時間**運作，找出 memory leak 等慢性問題 |
| 吞吐量 Throughput | 以**每秒處理的交易數或請求數**（TPS/RPS）為單位 |
| Virtual User (VU) | 壓測工具模擬出來的使用者，用以產生**可控負載** |
| 測試環境原則 | 應盡量**貼近正式環境**，否則數據參考價值有限 |
| 報告應包含 | VU、TPS、平均回應時間、**P95 回應時間**、錯誤率、資源使用率 |
| 執行本質 | **工程方法**，需要目標 + 情境 + 環境 + 分析四者兼備 |

---

## 測試類型比較

| 類型 | 負載程度 | 時間長度 | 目的 |
|------|---------|---------|------|
| Load Testing | 預期正常/尖峰 | 中等 | 驗證達到效能目標 |
| Stress Testing | **超過上限** | 中等 | 找崩潰點 + 恢復能力 |
| Soak Testing | 中等負載 | **長時間** | 找 memory leak 等慢性問題 |
| Spike Testing | 瞬間極高 | 短暫 | 驗證突發流量反應 |

---

## 效能測試報告必備指標

```
VU（虛擬使用者數）
TPS / RPS（每秒交易/請求數）
平均回應時間（Avg Response Time）
P95 / P99 回應時間（百分位數，比平均更有意義）
錯誤率（Error Rate）
資源使用率（CPU / Memory / I/O）
```

> ⚠️ 只看平均回應時間容易被誤導，P95 才能反映大多數使用者的真實體驗。

---

## 常見地雷

- 測試環境規格小於正式環境時，**不能**用倍數線性換算
- 測試環境資料量影響效能，不能用少量假資料代替
- 工具會用 ≠ 懂效能測試（k6 / JMeter / Gatling 只是工具）
- 用錯誤數據誤導團隊比不做測試更危險
