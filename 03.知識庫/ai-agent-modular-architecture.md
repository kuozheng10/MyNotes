## 核心論點

AI Agent 失敗不是因為模型不夠強，而是**把所有能力塞進同一個黑盒**。
真正能進工作現場的 Agent，靠的是架構分層 + 能力模組化 + 流程可重用。

---

## 四層架構

| 層 | 職責 | 常見錯誤 |
|---|---|---|
| **核心能力層** | 模型呼叫、狀態管理、工具調度 | 綁死特定 UI 或場景 |
| **工具外部能力層** | 插拔式工具，可依部門/安全需求配置 | 工具與核心邏輯混在一起 |
| **工作流程層** | 可重用的任務配方、SOP、workflow | 流程藏在 prompt，無版本無法測試 |
| **互動介面層** | CLI / Web / 桌面，只是同一套能力的入口 | 把核心邏輯放在 UI 層 |

---

## 模組化 vs. 追求更強 Agent

**讓模型更聰明，不如讓架構更乾淨。**

沒模組化的代價：
- 換模型 → 影響工具調度
- 增加工具 → 改動前端交互
- 客製化 → fork 整份產品

模組化的好處：
- 模型可替換，不必重寫產品
- 工具可增減，不必動核心邏輯
- 流程可複用，不必每次從 prompt 重組
- 組織可客製，不必走大分叉

---

## Workflow 是資產，不是臨時 Prompt

把任務邏輯藏在 prompt → 沒版本、難審查、難測試、不知哪步出錯

**正確做法：把高頻任務抽成明確模組**
- 定義輸入/輸出、前置腳本、AI 審查步驟、風險判準
- 適合場景：上版前風險檢查、文件審核、程式碼影響分析、資安稽核

> 流程結構化後，AI 才從「即興助手」變成「流程執行者」

---

## 權限與風險邊界

Agent 一旦能讀檔/改檔/呼叫外部服務，就不再只是內容工具。
「沒人敢讓它真的做事」的根本原因：**風險沒有被制度化管理**。

需要明確定義：
- 什麼操作需要用戶批准
- 什麼工具屬於高風險能力
- 失敗時如何中止、回復、追蹤

---

## 實務建議順序

1. **先定義核心工作單位**（任務能否拆成固定步驟、I/O 能否標準化）
2. **工具能力外掛化**（能力接口分離，工具與流程不耦合）
3. **高頻任務抽出為流程資產**（重複 2-3 次就值得抽）
4. **最後才做多介面部署**（核心穩定後才決定 CLI/Web/桌面）

---

## 更大意義：組織知識工程

工具可插拔 + 流程可重用 + 規則可審查 → 把**隱性工作知識**沉澱成**顯性可維護的系統資產**

AI 的真正價值不是自動化，而是讓資深員工腦中的判斷流程變成可執行的操作結構。
