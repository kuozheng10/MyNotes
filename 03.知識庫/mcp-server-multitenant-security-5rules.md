## 白話文先講：這篇在講什麼

想像你開一間出租倉庫，很多公司都在裡面租格子放貨。你請了一個管理員（MCP server），他手上有**萬能鑰匙**，能開任何一間倉庫。現在有個客戶打電話說「幫我查一下我的貨」，管理員該做的是：先確認這通電話是哪間公司打來的，只開那間公司的格子。

問題是：如果你用 AI 幫你訓練這個管理員，AI 通常只會教他「怎麼開鎖、怎麼查貨」，沒人特別交代他「打電話來的人是誰、只能開哪一間」——因為你在測試的時候，一直是你自己（同一間公司）在打電話，管理員每次都開對，你以為他訓練得很好。**其實他從來沒被逼著證明過「別人的格子他不會開」。**

這就是這篇文章的核心警告：**AI幫你寫的MCP server，「能跑、資料也對」不代表「安全」**，因為你的測試從頭到尾只用了你自己的身分測，永遠不會觸發「別人的資料被看到」這種bug。

## 這個問題的正式名字：confused deputy（被搞糊塗的代理人）

一個「有權限的程式」被騙去替「沒權限的人」做事。MCP server 天生就是這種角色——它拿著能存取整個資料庫的萬能鑰匙，代替跟它對話的使用者去查資料。它有沒有把查詢**限縮**在「這個使用者該看的範圍」，就是安全跟不安全的分界線。

## 五條規則（原文作者實際維護一個多租戶短網址服務 toui.io 的經驗）

### 1. 不要讓AI直接傳「編號」進來查資料

❌ 危險做法：AI傳 `get_stats(campaign_id: 42)`，後端要自己記得檢查「42是不是這個人的」，只要有一個地方忘記檢查，別人的資料就被看走了

✅ 安全做法：工具只收「名字」或「代號」，後端**在這個使用者的範圍內**自己去找對應的那筆，找不到就當作不存在

一句話：**你接受什麼當輸入，你就得負責檢查什麼**。在使用者範圍內解析出來的名字，天生不會變成漏洞。

### 2. 每筆查詢都要帶上「這是誰的」這個條件，而且查無資料要老實說「查無此物」

```sql
-- 危險：先撈出來再回頭檢查是誰的（容易忘記第二步）
SELECT * FROM campaigns WHERE id = ?;

-- 安全：從資料庫查詢本身就限定範圍，撈不到就是撈不到
SELECT * FROM campaigns WHERE id = ? AND team_id = ?;
```

還有一個細節：如果查到別人的id，要回「查無此物」，**不要回「你沒有權限（403）」**。403等於告訴對方「這筆資料存在，只是不給你看」，對方可以拿這個線索去試探哪些id是有效的（枚舉攻擊）。回「查無此物」，對方連「存不存在」都問不出來。

### 3. 「這個請求算誰的」只能從 token 判斷，絕對不能讓對話裡的參數決定

如果「這是哪個團隊的請求」是AI在對話裡傳進來的參數，那任何人只要改一個數字就能假冒別的身分——等於完全沒鎖。正確做法：身分永遠只從**登入憑證（token）**去解析，工具邏輯完全不去碰對話裡傳來的參數當作身分依據。

### 4. 長效期的授權（token）要能「立刻」撤銷，不能等它自然過期

有些token是每次都重新驗證身分的（短命），這種天生安全；但有些授權（例如你登入一次後給的長效期token）可能撐好幾個月，這時候如果這個人被停權/被踢出團隊，他手上那張「格式正確、還沒過期」的token還能不能用？如果系統只在「發token的那一刻」檢查過身分，答案就是「還能用」——這是個漏洞。正確做法：**每一次請求都要重新確認這個人現在還是不是有效使用者**，不能只信任token發出去那一刻的狀態。

### 5. 測試資料裡一定要種「第二個使用者」

這是最便宜、最容易被忽略的一條。如果你的測試環境從頭到尾只有一個使用者、一個團隊，那所有「跨租戶洩漏」的bug都測不出來——因為根本沒有「別人」讓你不小心越界。解法：在測試資料裡放第二個假使用者，寫測試專門「用A的身分去存取B的東西」，斷言「應該查不到」。

## 收尾原則

把「誰在呼叫」跟「他能看到什麼」焊死在後端，不要相信任何從對話請求裡傳進來的東西。動手寫之前先問自己：**換成別人的身分來呼叫，這裡會發生什麼事？**

## 這跟派哥有沒有關係？

**目前你自己用的 MCP server 都是單人使用（你自己一個人的 bank-sqlite、portfolio-sqlite、Gmail MCP、Notion 等），沒有「多租戶」問題** ——這五條規則的核心場景是「多個不同使用者共用同一個 MCP server，彼此資料要隔開」，你現在沒有這個情境，不用緊張。

**什麼時候會變得跟你有關**：如果你以後想把 investment-dashboard、cc_processor 這類工具開放給別人用（不只是你自己），或者做一個給多人用的服務，這五條就是必讀的檢查清單——尤其是第5條「測試資料裡種第二個使用者」，這個習慣現在就可以養成，之後不管做什麼多人系統都用得到。
