# AI Agent 來了，SaaS 還有未來嗎？— a16z「Headless Software」對談摘要

> 消失的可能不是 SaaS，而是我們使用 SaaS 的方式。

過去談 SaaS，關注的是使用者介面、操作流程，以及每個月要付多少席位費。但當 AI Agent 可以直接透過 API、MCP 或聊天介面讀取資料、執行任務，人類還需要登入那些複雜的後台嗎？這支 a16z 對談的核心觀點：不是軟體消失或沒有消失這麼二分，而是企業和管理的因素仍然存在，AI 時代軟體商真正的價值也在其中。

**最喜歡的一個例子**：新創公司看企業軟體時，常常是用新創的規模去思考——覺得公司只有 40 人，報帳這種事找一個人翻一翻收據就搞定了，或乾脆拍照 OCR 分類就解決。但公司規模到一萬人、橫跨 20 個國家、面對不同國家法律和差旅政策時，這種想法完全行不通，疊加上企業內部政策，整套系統就被編碼成這家公司獨有的樣子，沒辦法被取代。

---

## TL;DR

這支 a16z 對談提出了一個更關鍵的問題：消失的可能不是 SaaS，而是我們使用 SaaS 的方式。這也是 Headless Software 開始受到討論的原因（例如 Salesforce 推出的 Headless 360，是將傳統 CRM 的功能層（客戶資料、商機、自動化工作流）與 GUI 完全解耦，轉化為全面開放的 API、MCP Server、CLI 腳本命令）。

所謂 Headless Software，不是把軟體「拿掉」，而是把使用者介面從產品核心的位置移開。未來使用者可能不再親自登入 Salesforce、SAP 或 Workday，而是直接在 Slack、聊天視窗，甚至透過一個 Agent，要求系統查資料、產報表、更新紀錄或執行流程。對 Agent 來說，它不在乎按鈕放在哪裡、選單要點幾層，也不需要漂亮的 Dashboard。它真正需要的是底層資料、API、權限、商業邏輯，以及這套系統多年累積的規則。

所以 Headless 並不代表企業軟體變得不重要，反而可能讓我們更清楚看見，企業軟體真正的價值從來不只是 UI。像 SAP 之所以難以被取代，不是因為畫面做得多好，而是它裡面封裝了企業的財務規則、採購流程、權限設定、法規要求，以及大量客製化的例外處理。這也是為什麼「Postgres 加 API 就能取代 SAP」這種說法太過簡化——資料庫可以存資料，但它不會自動理解一家公司是怎麼運作的。

AI Agent 確實可能讓 SaaS 的介面退到背景，但資料、邏輯、權限與例外處理，仍然必須存在。因此 SaaS 的未來，可能不是消失，而是從一套「給人操作的軟體」，逐漸變成一套「讓 Agent 存取與執行的基礎設施」。

## 開場摘要

Sinofsky 說，讓軟體黏著的原因很多，但很大一部分跟人類使用軟體的方式有關。在 agent 的世界裡，真的還需要那個介面嗎？他認為資料、邏輯，以及底層儲存的一切，才是價值真正所在的地方。

Sinofsky 接著說，現在有一種對「vibe code 就能做出企業軟體」這件事的嚴重低估。他提到 Oracle 的 Larry Ellison 曾經開砲，痛批企業軟體很蠢，因為每個人都在客製化它。Sinofsky 說，一旦把最平凡的事情自動化，以為全部搞定了，全新的問題就會冒出來。

Amble 也點出一個常見的誤解，很多人以為只要有 Postgres 資料庫加上 API，就可以取代 SAP，這完全不是事實。她說，SAP 裡面封裝的邏輯和其他一切，遠比「這些資料剛好放在這個資料庫裡」重要得多。Sinofsky 補充，科技轉型發生時常見的現象是，沒有人能在指數成長正在發生的當下真正理解它。

## 什麼是「Headless Software」

這篇討論的起點，是 Seema Amble 大約一個月前寫的文章《Is Software Losing Its Head?》，緣起於 Salesforce 宣布走向 headless 化。

Seema 說 headless software 不是新詞，但最近公眾討論明顯升溫。關鍵新聞點是 Salesforce 推出的 Headless 360——很典型是 Salesforce 一貫的操作，與其說是產品突破，不如說更像行銷宣傳。實際上什麼都沒變，360 產品用的還是一直存在的那些 API，只是重新包裝成 360 這個品牌。但這個宣布確實承認了正在發生的事實：傳統軟體是圍繞著人類存取而建構的，用來捕捉資料的是工作流程，而在 agent 的世界裡，UI 之所以不重要，是因為 agent 根本不是透過 UI 存取軟體的。headless 的核心在於：資料、邏輯，以及底層儲存的一切才是價值所在，而不是最上層被追蹤的工作流程軟體。

Seema 舉了 Notion 的 headless 產品當對照——這個案例反而更合理，因為 Notion 使用者普遍技術能力更強，更習慣用 agentic 的方式建構東西；相較之下 Salesforce 使用者技術能力沒那麼強，雖然也有愈來愈多人在用 Salesforce 打造 agent。

Elena 補充，這個概念也可以套用在像 chatbot 這麼簡單的東西上——Salesforce 收購了 Slack，現在可能就是透過 chatbot 跟 CRM 互動這麼簡單。Seema 提到讀過的數據：**Slackbot（Slack agent）使用量成長了 300%**，代表使用者根本不需要登入 Salesforce 介面查資料。

Steven 說現在正處於一種「定義地獄」，但這件事很重要。他認為「agent」這個詞，其實是「一個跑很久、可能還跑不完的程式」的新說法——半開玩笑地說，把跑很久的程式改稱為 agent，是史上最厲害的行銷包裝，因為以前這種程式叫做 bug，現在卻變成最酷的新功能。

**Steven 把 agent 在做的事拆成三種**：
1. **查詢（look something up）**：相對輕量，所有系統都做得不錯。很多新宣布的 headless agent API，其實就只是查詢，換了介面做以前就能做的事。
2. **執行（do something）**：牽涉到冒充特定人、用他們的憑證、席位怎麼算這類企業軟體特有的複雜問題，只要想真正改變系統記錄，全都會冒出來。
3. **分析（analyze）**：比查詢複雜得多，通常要動用多個系統。非常適合 agent（不受時間限制、可反覆迭代、可跨模型比對結果），但也是幻覺問題最嚴重的地方——因為必須有辦法驗證分析過程每一步都正確。

Steven 認為，headless 跟 agent 常被混為一談，但必須先搞清楚自己在講哪一種，因為在學習曲線和部署進度上，大家其實站在這三種情境的不同位置。

## 什麼讓企業軟體黏著

Seema 說，讓軟體黏著的原因很多，但很大一部分跟軟體是圍繞著人類的互動方式打造的有關：讀寫次數、存取頻率、下游工作流程，還有所有沒被寫下來的 SOP，都內化成肌肉記憶和流程，甚至外部單位也牽涉其中。CRM 黏著，是因為業務代表每天進出、已經習慣；財務靠它出帳、行銷也依賴它。企業需要單一真相來源（single source of truth）——加上稽核要求，造就了黏著性。

Steven 補充，最黏著的事情其實是**向客戶收錢**。一旦在收錢，客戶要停止付錢非常難。深入問為什麼黏著，答案取決於問的是公司裡的誰：合規人員說是規範要求、系統管理員談新用戶上手、使用者談肌肉記憶和按鍵習慣。最重要的還是先把軟體賣出去，之後黏著的原因會變成各種贏家故事。

**Outlook 案例**：任何試圖取代 Outlook 的人，很快會發現「代理存取權限（delegate access）」和「行事曆可以被多人擁有」有多重要。從來沒有一場會議專門討論怎麼讓行事曆變黏著，但事後發現，通用汽車不會因為行事曆這種小事放棄 60 萬個席位。

Elena 總結兩個重點：**慣性（inertia）**是非常強大的力量；沒有人打造軟體時照清單打勾，但軟體觸角延伸到整個組織、深植日常、已經付費很久，就會滲透進人們做事的方式裡，變得難以連根拔除。

## 軟體末日？為什麼 SAP 可能不會消失

Steven 說，唯一比 SAP 更黏著的軟體，是藏在幕後的軟體——比如保險公司 50、75 年前寫的系統，根本沒辦法被取代（企業找 COBOL 工程師維護每個州的保險軟體）。

他認為 **Stripe** 是目前最成功的案例之一，因為 Stripe 頭一次真正解決了「向人收錢」這個系統——整合每個國家、司法管轄區、跨境交易、貨幣兌換的稅法，過去從沒有人真正解決過。

如果把 SAP 從一家大型汽車製造商裡拿掉，這家公司基本上就不存在了；沃爾瑪也一樣——因為公司把商業規則編碼進了這個產品裡，不只是購買或使用它。

Seema：**「Postgres + API 就能取代 SAP」是個誤解**。SAP 裡封裝的邏輯遠比「資料剛好放在資料庫裡」重要得多。SAP 導入要花好幾年，不只是系統整合商慢，而是這套系統被客製化成符合企業實際運作方式——這是為什麼不能把軟體完全抽象成「資料庫加 API」。

**新創公司常見的規模誤判**：公司只有 40 人，報帳一個人翻一翻收據就搞定，或拍照 OCR 分類解決。但公司到一萬人、橫跨 20 個國家、面對不同法律和差旅政策，這種想法完全行不通，疊加企業內部政策，整套系統就被編碼成這家公司獨有的樣子，沒辦法被取代。

**Larry Ellison 的 80% 解決方案**：1990 年代末，Oracle 創辦人痛批企業軟體因為每個人都要客製化而很蠢，主張企業應該用「80% 的時間能運作」的方案就好。Steven 說當時企業界的反應：第一，這根本是在幫自家軟體打廣告，因為 Oracle 軟體本來就只能做到 80%；第二，事情根本不是這樣運作——福特、豐田、通用、戴姆勒的差異，不只是看同樣畫面，而是選擇看哪些畫面、做了哪些客製化，這一切都在 SAP 裡運作。

**Excel / 高盛的故事**：Steven 早年向高盛推銷 Excel（比 Lotus 1-2-3 好用），對方回：「我不覺得你們懂，我們靠 Excel 賺的錢比你們還多」——因為高盛把 Excel 用出了極高的差異化程度：自己開發外掛、寫大量程式碼、定義自己的工作方式。

**Vibe code 做企業軟體的低估**：某成長期新創（約千人規模）的 revops 主管想重建整套 Salesforce，一開始覺得反正知道所有欄位、匯進去就好。Steven：真正困難的不是這個，而是決定哪些東西要被記錄、組織架構界線怎麼劃、系統之後由誰長期維護。Vibe code 出的 CRM 早晚會過時沒人碰，因為維護很痛苦。

Elena 補充：現在有一整個新創生態系建立在 SAP 之上，圍繞著讓人頭痛的事情打造產品，骨子裡依賴 SAP——老牌系統已深植到根本無法撼動，新一代挑戰者只能在周邊建構，而非要求客戶完全遷移。

## 易用層：AI 讓資訊「送到面前」而不是「跑到 UI 裡查」

Seema：很多 AI 應用其實是在解決「怎麼讓一切更對話式、更容易取用資訊」的問題——不用跑 SQL 查詢或翻好幾個畫面，能否跨表格、跨地區用自然語言快速查詢、自動產生客製化報表。存取 UI 已經變成可有可無的選項。

Steven 認為這是最值得帶走的觀點：**企業軟體幾乎總能做到使用者想要的事，使用者只是不知道怎麼讓它做到**（沒有 SAP 生不出來的報表，只是不知道怎麼操作，或權限沒設好）。換個角度：企業軟體裡最常被用的兩個功能，其實原生不存在於任何企業軟體裡——**匯出成 Excel** 和 **匯出成 CSV/PDF**。這是逃生閥，讓使用者做原本系統做不到的分析。現在語言模型能用簡單得多的方式消化這些匯出檔案（例如把 20 份 PDF 丟進模型做例外分析）。

**這些臨時性的商業流程（ad hoc business processes）往往變成最有趣的東西**——因為這正是企業實際運作的方式，也是下一批新產品的原型。CRM 以前也不過是一張 Excel 試算表，後來 Siebel（SAP/Salesforce 前身）才做成一整間公司。

Seema：Salesforce 其實是業務團隊的強制機制，確保蒐集到需要的資訊。Agent 世界裡，agent 不太在乎欄位怎麼排列，但仍需要能存取資訊，以及「情境（context）」——也就是各種例外狀況怎麼處理、邊緣案例、權限設定，以及沒有被記錄在欄位裡的政策（例如對象在亞洲 vs 美國，回應方式不一樣）。這些存在某個人的腦子裡，才是 agent 真正需要的情境。

## 例外處理才是整場遊戲的核心

Elena：企業裡幾乎所有有趣的事情，都是例外狀況（比喻：麥當勞點餐機永遠處理不了「兩種口味混一杯」的 McFlurry；企業定價永遠是「打電話問我們」）。

Seema：例外目前根本沒被系統性記錄下來。現在可以用語音 agent 蒐集合規檢查電話裡的例外，或用「電腦操作型 agent」觀察人類怎麼點擊、怎麼回應，用 LLM 處理這些資料，開始蒐集情境。但這需要時間——銷售週期很長，例外不會以立刻取得資料的頻率出現，必須累積足夠多互動才能真正理解。

Steven：**亞馬遜**是很好的觀察對象——真心不想讓人類介入客服。chatbot 理解收到錯誤商品，直接寄新的給你，還會用這些資料改善出貨、處理、倉儲流程。AI 能力真正有趣的地方，在於推動企業對「怎麼處理例外」產生不同定義和行為——等走過 1.0 階段，人們會安心讓 AI 做決定，因為這帶來更高的可預測性和可重複性。（Elena 反向觀點：短期內客服體驗可能反而變差，因為不再自動站在客戶那一邊，得重新為自己的立場辯護。）

Seema 補充：**權限管理（permissioning）**也是長尾的一部分——隨著給人或 API 存取權，必須決定什麼情況能擷取資料、什麼時候能寫入。理想上只有一個中央資料庫作為真相來源，但多個對象要存取寫入時，誰能在什麼時候存取就成了額外問題。

## 生產力創造新場景，而不是減少工作

Steven：科技轉型時常把指數成長誤判成線性成長去外推。一旦某件事能被自動化變輕鬆，人們就會構想出一大堆全新的事要做——**亞馬遜淘汰電話客服後，發明了「壞掉的消耗品乾脆讓客戶丟掉、不用寄回」**（因為可能已污染，直接讓客戶丟掉更便宜）。但後台需要不斷分析怎麼避免同樣的事再發生——長尾沒有變短，只是用不同方式變長了。

Elena：這正是創新的本質——餅在變大，不是固定不變。法律產業：AI 不會讓律師消失，而是合約會變得更長更複雜，涵蓋更多情境，反而帶動更多訴訟，創造新生態系。放射科：AI 工具很受歡迎，但反而出現人力短缺（相關性非因果）。

**報帳的演化案例**：一開始沒系統 → 試算表處理 → 分析報帳資料的系統 → 出現「商務差旅」這個新領域（用里程數優化、把差旅導向最優惠價格而非固定航空公司）→「商務差旅分析」這個更大範疇，需要的人力遠比訂機票多 → 分析本身又推動新流程和行為，變成企業間的差異化因素 → 最後演變成「遠端工作優化工具」，完全不同的東西。

Seema：物理世界和數位世界始終並存——業務仍會親自談成交易、人們仍會搭飛機出差。永遠會有需要捕捉並優化的**資料排放（data exhaust）**，這件事不會消失。

Steven 用開源軟體開發當例子：軟體開發最難的事，是必須在某個時間點宣告「完成」，這件事沒有 API 可以做——需要一群人達成共識。但這群人卻會說像結算財報這種商業流程「應該」直接變成一支 API，這在心智模型上其實是同一件事。商業真正的核心，就是人們在做決定，軟體只是提升層次、抽象化，改變決定的內容、方式和工具。

Elena 補充：最好的做法是把一切記錄下來（合成式蒐集，不是全面監控）。呼應她的觀點：專業知識存在於組織雲端之上，是尚未被開發的資源（Box 的 Aaron Levie 談散落公司各處的 Word/Excel 文件裡的資產）。AI 是第一個真正能挖掘公司裡這些非結構化資訊的技術。

## 歷史的重演：從微軟反壟斷案到 MCP

Elena 提到 Steven 去年寫過一篇文章，把 MCP server 的興起連結到早年司法部對微軟的反壟斷訴訟（微軟很多產品被歸類為「中介軟體 middleware」）。

Steven：MCP 真正的核心問題，很大程度是工程師對「什麼是好的軟體架構」的想像所驅動，卻很少考慮世界的實體現實要如何無縫運作。工程師希望每個工具都有乾淨的 API，但**沒有任何軟體會甘願被上層另一層系統架空**——沒有人想被告知「你的工作就是儲存報帳資料的 SQL 格式，其他什麼都不用做」。客戶也不想從一堆不同供應商拼湊使用情境，因為系統穩定度會等同於最不穩定的那個環節（報帳軟體公司倒了你就束手無策）。反過來，那些軟體公司也不會坐以待斃，會去做客戶正在用自家產品做的事情（SAP 自己也在做，雖然大多做得不怎麼樣）。**這個中介軟體層永遠不穩定**——畫在 OSI 七層架構圖上很漂亮，但實際上從沒真正穩定過。

Seema 補充：Workday 一直有 API，但你無法用乾淨方式把所有資料完整擷取出來、完全不用 Workday 就能運作——因為公開一切等於把自己變成笨資料庫，沒有誘因這麼做。

**Seema 提出的三條路**：
1. **直接用既有巨頭的 agent**（如 Salesforce Agentforce，把 Salesforce 當純後端）——她不看好，因為 Salesforce 不會甘願只當背景資料庫，對既有軟體巨頭能做出優秀 agent 並不樂觀。
2. **完全自己動手做（DIY）**——控制權最大，但等於重建真正的企業軟體，難度極高（「一邊做心臟直視手術，一邊病人還活著」）。
3. **Agent 持續被建構、資料在背景持續被吸收整合**（a16z 投資 AI 軟體的原因）——跟 SAP 並肩運作或疊加可視化層，強化業務使用者體驗，同時不丟掉背景累積的邏輯。語音 agent、錄音、文件轉錄都在蒐集新的系統記錄——或許有一天這些 AI 新創會取代後端系統記錄，但是用有系統的方式、透過觀察企業實際運作方式達成。

## 新創公司現在最大的機會在哪裡

Seema 認為機會在於做既有巨頭沒在做的事——**從單純蒐集資料，進化到怎麼在資料之上採取行動**：
- CRM 不只記錄通話，而是判斷該優先處理哪些名單、哪些客戶有流失風險、主動發送陌生開發訊息；建立 agentic 循環（發訊息→看回應→理解哪些做法有效），蒐集基準資料（不同地區該用什麼開場語氣）。
- 為**實體世界**打造的垂直軟體（建築業、製造業）蘊藏過去很難被捕捉的資料，需要持續整合 agent 已能操作的軟體資料 + 人類和機器現場實際做的事。

Steven：企業軟體有一條放諸四海皆準的道理——**最困難卻也最愚蠢的做法，是試圖跟既有品類正面對決**（包含用完全一樣的方式做）。最大的機會永遠是看既有品類的心智地圖，卡在兩個既有玩家之間——因為劇烈技術轉型時，既有玩家絕不會打亂自己的產品線和市場策略，只會把 AI 硬掛在既有產品上、硬撐過這波風暴。新創的機會就是瞄準中間地帶，用全新方式做事，不用先完成客戶要求的 8000 件事，而是回答一個自己能掌控的問題：你存在的理由是什麼。

**HTTP/HTML 例子**：Web 之所以能取代 client-server，不是因為做到了 client-server 能做的所有事（事實上一件都沒做到），而是用全新方式實現了那個概念——儘管傳統廠商在 client-server 上已投入一兆美元。

Seema 補充：機會不只在兩個傳統廠商之間，也在**企業內部兩個不同職能部門之間的翻譯層**（情境）——軟體一直賣給單一部門，但部門交接地帶（帳單、交易這類跨部門事務）也蘊藏機會。

## 企業軟體的網路效應

Elena 問：企業軟體是否會出現網路效應（消費端軟體常見的護城河）？

Steven：公司之間（外部）的網路效應因合規和安全理由非常困難，但**企業軟體最大的網路效應發生在公司內部**，現在正透過 chat 這種形式看到——多數人不特別想優化自己的工作流程，只求別把事情搞砸，但總有一小群人（像那些先用 Excel 的高盛銀行家）想用更聰明的方式談成更多交易。

Steven 分享：SAP 朋友寫白皮書時，他問對方想回答什麼問題、寫個 prompt 就生出白皮書——啟動了對方團隊裡類似病毒式擴散的循環，因為大家看到怎麼讓工作變輕鬆，而且觸手可及。

**一個能讓過去無法溝通的兩個職能部門開始對話的工具，是非常珍貴的機會**——這正是企業軟體整合的本質（過去得靠 Accenture 這類顧問公司人工蠻力）。Figma 在設計和產品開發之間做了很多這樣的事；IT 和財務部門最終會用上同時幫雙方做預測的工具，雲端運算促成了這件事。Steven 認為，運用 AI 把組織裡原本不會溝通的部門連結起來的軟體，會是一個全新的品類，這正是一個巨大的機會所在。
