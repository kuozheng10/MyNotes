# 通知型 AI SaaS 開發範式（以機票價格為例）

> **核心洞見**：「通知型 SaaS」是最適合 AI 時代的第一個產品模型。
> 架構固定、邏輯清晰、可重複用在任何「盯價格/庫存/狀態」的場景。

---

## 通知型 SaaS 的標準架構

```
用戶設定條件（航線 + 目標價）
        ↓
背景排程持續抓資料（Cron Job）
        ↓
AI / 規則判斷：符合條件？
        ↓
    是 → Email / TG 通知
    否 → 等下一輪
```

### 六個必做模組

| 模組 | 說明 | 常用技術 |
|------|------|---------|
| Landing Page | 產品介紹、轉換 | Next.js + Tailwind |
| 登入系統 | 用戶帳號 | NextAuth / Clerk |
| 資料庫 | 儲存條件、歷史 | PostgreSQL / Supabase |
| 背景排程 | 定時抓資料 | Vercel Cron / launchd |
| 通知 | Email / TG / Line | Resend / Mailgun |
| 付費 | 訂閱制 | Stripe / ECPay |

---

## 為什麼「通知型」特別適合作為第一個 SaaS？

1. **MVP 快** — 核心功能就是「爬資料 → 比較 → 發通知」，一週可完成
2. **留存率高** — 用戶設定後就不用管，但每次收到通知都想到你
3. **付費邏輯簡單** — 「免費版每天查一次，付費版每小時查一次」
4. **可套用在任何垂直市場** — 換資料來源就是新產品

## 技術注意事項

- **反爬問題**：部分網站有反爬，需要 Playwright headless 或付費 API
- **排程頻率**：免費版 Vercel Cron 每天一次；需要更頻繁考慮 launchd / Railway
- **通知限制**：Resend 免費 3000 封/月，超過要付費
- **Stripe 台灣**：需要公司帳號，個人可用 ECPay 或 Paddle

## 派哥下一個 SaaS 建議

### 🏆 第一推薦：哩程票提醒 SaaS

**為什麼最適合派哥：**
- 自己就是目標用戶（最懂痛點）
- 台灣玩哩程的人持續增加，有市場
- seats.aero 有 API，資料來源穩定不用爬蟲
- 技術棧就是這篇文章的範本，可直接套
- 差異化：專注哩程票（不是一般機票）+ 中文介面

**最小 MVP：**
設定出發地 + 目的地 + 艙等 + 哩程預算 → 有票自動 TG/Email 通知

**現有資源可用：**
- My Wallet Trip 前端技術直接轉用
- launchd 排程 know-how 已有
- 個人需求：2027/5 商務艙日本 3 人票還在追

---

### 🥈 第二推薦：信用卡優惠雷達

**為什麼：**
- cc_processor 已有各卡片的解析 know-how
- 台灣信用卡優惠資訊分散、難整合
- 可以做「設定你的卡片 → 自動彙整符合你消費習慣的優惠」

**挑戰：** 各銀行網站反爬，需要 Playwright 或人工維護

---

### 🥉 第三推薦：Claude Code 效率工具

**為什麼：**
- 全球市場（英文 SaaS，不限台灣）
- 自己最懂痛點（CLAUDE.md 生成、token 追蹤、skill 管理）
- 競爭少，使用者群體在快速成長

## AI SaaS 的核心競爭力

> AI 不缺會寫程式的人，AI 缺的是能把程式變成產品的人。

技術是門檻，但真正的護城河是：
1. 對垂直市場的深度理解（知道用戶痛點在哪）
2. 快速迭代能力（Claude Code 讓你比競爭者快 5~10x）
3. 分發渠道（社群 / 口碑 / SEO）
