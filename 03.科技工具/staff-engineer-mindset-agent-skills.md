## 這是什麼
這篇文章介紹如何將資深或主任工程師（如前 Meta 主任工程師 Lauren Tan）的技術調查與程式碼理解邏輯，轉化為 AI Agent 的「技能（Skills）」。透過這種方式，能讓團隊成員在探索複雜問題時，都能運用與專家相同的思考模式來搭配 AI 代理。

## 核心概念
- **代理技能化 (Skill-based Agency)：** 將專家的直覺與 SOP 封裝成 Agent Skills，使 AI 能夠遵循高品質的調研路徑。
- **深度實作導向：** 不憑函式名稱猜測功能，堅持使用 Glob（尋找檔案）、Grep（搜尋關鍵字符）、Read（理解實作內容）的三步驟進行實測。
- **全貌調研流程：**
    1. **找出進入點：** 釐清行為的觸發源（API、使用者或排程）。
    2. **追蹤呼叫鏈：** 讀懂每個函式，觀察資料流動與轉換。
    3. **整理核心抽象：** 理解核心型別、介面、類別存在的目的。
    4. **界定系統邊界：** 釐清子系統間的輸入輸出關係。
    5. **識別陷阱：** 注意歷史遺留程式碼或易誤解之處。
- **誠實的探索報告：** 在無法百分之百確定邏輯連接時，應明確標註而非含糊帶過或產生幻覺。

## 使用方法 / 快速啟動
1. **分析流程：** 啟動 AI 代理後，要求其先進行代碼庫掃描（Glob）與關鍵邏輯搜尋（Grep）。
2. **深度閱讀：** 強制 AI 讀取具體實作，不接受基於名稱的概略分析。
3. **產出結構化描述：**
    - 描述觸發機制。
    - 描繪資料從 A 到 B 的轉換邏輯。
    - 列出核心型別定義與其關聯。
    - 指出可能的技術債或不直觀點。
4. **驗證完整性：** 檢查 AI 是否能完整描述全貌，若有斷點則需進一步深入追蹤。

## 對派哥的啟示
- **技能文件化：** 派哥目前開發的 `openclaw` 專案中，應將各類自動化工作流（如 `summarize-url`）的「調研階段」標準化，編寫成 `SKILL.md`，讓 Gemini CLI 在執行指令前先依循主任工程師的邏輯進行環境分析。
- **針對性調研：** 派哥在台灣處理各種自動化 API 串接（如 Telegram Gateway 或 Notion API）時，可以建立一套「異常調研技能」，讓 AI 在遇到錯誤時自動執行「進入點 -> 呼叫鏈 -> 邊界檢查」的標準動作，減少手動 Debug 時間。
- **知識工程的實踐：** 將此思維應用於「組織知識工程」中，協助台灣企業將內部資深工程師的隱性知識轉化為顯性的 Agent Skills，實現開發力的自動化擴張。
