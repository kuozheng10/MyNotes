## 問題背景：兩層缺口疊加

### AI 帶來的新缺口
- AI 幾秒生 200 行 code → reviewer 看不完
- AI 自己寫 code 又自己寫測試 → 全綠燈不代表品質
- 覆蓋率高，上線還是出包
- 出事了，責任怎麼追？

### 數據支撐
| 數據來源 | 數字 |
|---------|------|
| Stack Overflow 2025 | 84% 開發者已用或計畫用 AI 工具 |
| Veracode 2025 | **45% AI 生成程式碼含安全漏洞** |
| Apiiro 2025/09 | AI 程式碼導致特權提升漏洞增加 **322%** |

### 敏捷時代留下的測試債（舊問題未解）
- 發布頻率拉高 → 測試時間被壓縮
- 需求不清楚 → 測試憑感覺
- 回歸測試積累 → 跑一輪要好幾天

AI 把開發速度再往上推，舊問題沒解，缺口只會更大。

## AI 時代需重視的測試實踐

| 實踐 | 說明 |
|------|------|
| **Spec by Example (SBE)** | 需求用具體範例定義，測試才能當停止條件 |
| **Eval Gate** | AI 產出的驗收閘門，不讓未驗證程式碼進入 main |
| **對抗式測試** | 獨立 verifier agent 挑毛病，不讓寫的人自己驗收 |
| **Self-Healing CI** | CI 失敗自動修復並回報 |
| **Test Impact Analysis** | 只跑受影響的測試，加快 CI |
| **Audit Trail** | AI 修改的可追溯性，出事能追責 |

### 業界案例
- **Canva Magic Switch** — 大型 AI 功能的測試設計
- **Meta 預測式測試選擇** — 智能選擇要跑哪些測試

## 敏捷測試三層架構

### 測試左移
- 需求階段就對齊認知（SBE）
- 確認「對」長什麼樣子，再讓 AI 動手

### 開發過程多層次守住品質
- unit → integration → e2e 各層有自己的 guard
- AI 寫的測試需要人審查意圖，不只看通過率

### 測試右移 + 連續驗證
- 上線後的真實世界場景驗證
- flaky test 管理、canary 部署監控

## QA 角色轉變

```
過去：執行測試
現在：品質編排者（Quality Orchestrator）
```

- AI 接手越來越多測試執行工作
- QA 價值往「設計驗收條件」「管理 Eval Gate」「追蹤 Audit Trail」移動
- 需要的新能力：SBE 撰寫、AI 工具評估、風險判斷

## Eval Gate 深探：三層架構 + 雙信號線

> 來源：敏捷三叔公 LinkedIn（2026-06-22）

### 核心問題：LLM 沒有唯一正確答案

傳統測試：`assertEqual(A → B)`，答案唯一。  
LLM 測試：同一問題有十種合理答法，不能逐字對比。

Eval Gate 不驗「答案對不對」，驗的是**這個輸出有沒有違反我們在乎的規則**：
- 有沒有亂編沒根據的內容（hallucination）
- 該引用來源的有沒有引用
- 該拒答的有沒有拒答
- 語氣有沒有歪掉
- 該回答的有沒有答完整

### 三難困境：便宜 × 快 × 統計顯著

| 若只顧兩個 | 結果 |
|-----------|------|
| 便宜 + 快（30 個範例） | 誤差範圍 > 品質退步幅度，假警報多，團隊學會無視 |
| 統計顯著 + 快（每 PR 跑 2000 筆） | $9/PR，22 分鐘，一個月後被默默關掉 |
| 統計顯著 + 便宜（夜間才跑） | 合理但無法即時擋 PR |

### 業界三層實作

| 層 | 觸發時機 | 內容 | 成本 |
|----|---------|------|------|
| **L1：快速閘門** | 每個 PR | 格式、引用、hallucination，用小型分類器判 | 幾乎免費 |
| **L2：LLM 評審** | 每日夜間 | 最強模型對全部範例打主觀分（語氣、幫助性） | 集中跑，攤平費用 |
| **L3：人工分群審查** | 每日 5 分鐘 | 工程師只看「問題分群後的異常」，不看每筆 | 時間極低 |

### 雙信號線

- **絕對地板**：抓災難級崩壞（如客服 agent 拒答率暴增）
- **滾動基準線**：跟過去 7 天平均比，抓「慢慢漂的退步」

> 基準必須滾動更新，不能凍結在上線當天——模型會漂、prompt 會漂、資料也會漂。

### 典型失敗案例

工程師只改一行 prompt → CI 全綠 → merge → 12 小時後客服 agent 開始把正常問題判成「我無法回答」。沒有 function 壞掉，只是 LLM 輸出悄悄變差了。

## Self-Healing CI — 兩個層次 + 三級成熟度

> 來源：敏捷三叔公 LinkedIn（2026-06-22）

### 兩個層次（常被混為一談）

| 層次 | 痛點 | 解法 |
|------|------|------|
| **L1 自我修復測試** | UI 改版 → locator 抓不到 → 整批紅燈（程式沒壞，測試腳本壞了）| AI 同時看文字/位置/class/DOM 結構「指紋」，自動找回元素 |
| **L2 自我修復管線** | build 失敗、dependency drift、YAML 打錯、環境問題 | 回饋迴圈：偵測 → 分析根因 → 動手修 → 驗證 → **提 PR 給人審**（不直接 merge）|

**核心迴圈（跟 Harness/Agentic Loop 同一家族）：**
`偵測失敗 → 分析根因 → 動手修 → 驗證 → 提 PR 給人審`

### 三級成熟度

| 級別 | 名稱 | 行為 |
|------|------|------|
| ① | 觀察者 | AI 只在旁評分，不擋 build，先累積「什麼叫好」的黃金資料集 |
| ② | 守門員 | blocking gate，信心分數低於門檻就擋下來，但不動手修 |
| ③ | 治療者 | 有寫入權限，自動修測試腳本或設定檔，再提 PR |

> 大部分認真的團隊現在卡在 ① 到 ② 之間。

### LLM-as-a-Judge（測 AI Agent 本身）

當測試對象是 AI agent，傳統字串比對完全失效。用另一個專門模型給主 agent 的輸出評分 → 這就是 [[ai-testing-agile-quality#Eval Gate]] 的核心概念。

### 工具地圖

| 類型 | 工具 |
|------|------|
| 測試平台型 | Mabl、Testim、Functionize、testRigor |
| 視覺回歸 | Applitools、Percy、Meticulous |
| 框架加掛 | Katalon、ACCELQ、Virtuoso QA |
| Playwright 導向 | Shiplight、Playwright MCP、Stagehand |
| 管線層 | GitHub Actions + Copilot、GitLab CI、CircleCI、Jenkins、Dagger |

**技術路線差異：**
- Testim 類：ML 多維評分（文字+位置+class+id+結構）
- Functionize 類：NLP + 電腦視覺，DOM 完全改掉也能辨識（含 canvas 渲染 UI）

### 效果數字（打折看）

| 數字 | 來源 | 可信度 |
|------|------|--------|
| 腳本維護 -95%、回歸快 2× | 全球零售商案例 | ⚠️ 廠商 case study |
| 維護工時 -88%（3 個月）| 不明 | ⚠️ 廠商 case study |
| 維護工作節省 ~70% | Capgemini World Quality Report | ✅ 較中性第三方 |

**建議用「七成」而非「九成五」跟人溝通預期。**

## 為什麼 GenAI 時代下，RD 更需要懂測試？（課程預告文，2026-07-02）

> 來源：敏捷三叔公，同一系列課程「不確定時代下的敏捷測試」預告文

### 核心論點：瓶頸從「寫程式碼」轉移到「驗證程式碼」

以前瓶頸是寫 code，現在瓶頸是確認 AI 寫的 code 對不對。若 RD 不懂測試，AI 給的測試可能「看起來像真的」——會過、覆蓋率漂亮，但驗證的是錯的行為（測試界的 **oracle problem**：測試無法自己定義什麼是「正確」，只有懂需求/風險的人能定義）。

### 數據支撐（新增，跟本篇上方數據互補）

| 來源 | 發現 |
|------|------|
| DORA《State of AI-Assisted Software Development》(2025) | 90% 開發者用 AI；因交付變快、改壞重做比例惡化，**新增第五指標 rework rate（重工率）** |
| DORA《Balancing AI Tensions》(2026-03) | AI 加速產出，代價是沉重的「驗證負擔（verification overhead）」 |
| DORA《ROI of AI-Assisted Software Development》(2026-01) | **放大器效應**：AI 最大報酬來自組織/工程實踐本身的品質；基礎差的團隊用 AI 只會把生產力賠光在下游混亂 |
| METR RCT (2025) | 資深開發者用 AI 處理**熟悉**的 codebase，實測反而**慢 19%**（自己卻感覺變快）——慢在檢查/修正/確認 AI 產出 |
| GitClear | AI 普及後，複製貼上式重複程式碼上升，重構行為下降 |
| Veracode | AI 生成程式碼有相當比例帶已知安全弱點（模型學走了公開程式碼的壞習慣） |

### 五個「RD 不得不懂測試」的場景

1. **一人當一條龍，龍變更長**：沒有專職 QA 的團隊，AI 讓一人 sprint 產出過去半團隊的量，驗證責任全部回到自己身上
2. **模糊需求 → SBE 範例就是餵給 AI 的 context**：具體範例不只釐清需求給團隊看，也是 AI coding 工具對齊產出的依據（呼應本筆記上方 [[ai-testing-agile-quality#AI 時代需重視的測試實踐|SBE 實踐]]）
3. **驗證能力決定真實速度**：AI 時代的速度由「多快能確認程式碼可以上」決定，不是「多快能生出程式碼」；懂測試的人知道風險在哪，把驗證時間花在刀口上
4. **迴歸測試量變大、AI 測試品質也是問題**：AI 產測試常見兩種病——assert 太弱測不出東西、把現有行為（含 bug）當正確答案固化。RD 的價值在判斷哪些該自動化、AI 產的測試哪些是裝飾品
5. **AI 寫的 bug 長得跟人不一樣**：邊界條件、錯誤處理、安全預設值是 AI 最常省略的，要主動在 code review / prompt 階段設防線

### 與本筆記已有內容的關係

- 與上方 [[ai-testing-agile-quality#Eval Gate 深探：三層架構 + 雙信號線|Eval Gate]] 呼應：「你讓 AI 批改 AI 自己考卷」= 沒有獨立 verifier 的必然結果
- 核心共識不變：AI 是放大器，好的測試判斷力被放大成好習慣，差的判斷力被放大成技術債

## Spotify 案例：2,900 工程師、2,000 萬行 monorepo，verification 是核心瓶頸（2026-07-02）

> 來源：敏捷三叔公，整理自 Anthropic 官方訪談（Boris Cherny × Spotify 首席架構師 Niklas Gustavsson）
> 原始影片：https://youtu.be/9DHZLw5653E ｜ METR RCT：arXiv 2507.09089

### Spotify 自報數據（廠商自述，非獨立驗證）

| 指標 | 數字 |
|------|------|
| 工程師規模 | 2,900 人 |
| 每天部署次數 | 4,500 次 |
| Codebase 規模 | 2,000 萬行（monorepo） |
| AI 協助的 PR 比例 | 約 73% |
| PR 頻率提升 | 75%+ |
| 早期 LLM-as-judge PR 成功率 | 從 ~25% 拉到 80%（模型變強後這層 judge 已移除） |

內部工具 **Honk**：把 Claude Agent SDK 跑在 Kubernetes pod，可從 Slack 呼叫。最關鍵的能力不是生成，是 agent 能自己跑 CI build、看哪裡失敗、繼續修。

### 核心論點：checking ≠ testing，verification 才是真正瓶頸

Boris 與 Niklas 都強調：agent 自己做事、沒人逐步盯著時，「驗證」是系統裡最重要、也是多數公司投資最不足的一環。Spotify 是被逼的——常規改動改成自動合併後，「每個改動有 owning team 人工審」的前提直接崩了。

### 四個坑（收斂重點）

1. **品質只是「維持」，不是「改善」**：速度大幅提升，品質指標持平。自動化驗證做的是 **checking**——對既有 oracle 比對已知期望，擋得住「你知道會壞的東西」，擋不住「沒人想到要斷言的問題」。每天 4,500 次部署、多數 PR 沒人看就合併，是 **oracle problem 的硬核版本**
2. **AI 同時寫 code + test = 同義反覆**：agent 生實作又生剛好會過的測試，綠燈只證明兩者自洽，不證明符合需求。誰確認測試表達的意圖是對的？—— 這正是 **SBE（人寫的具體範例）** 的位置，機器自己生不出這個 oracle
3. **前置條件極重，不能照抄**：Spotify 工程部落格原話「你無法安全地自動化你不理解的東西」。撐起 Honk 的是十幾年 Backstage、元件目錄、ownership、標準化建置。自動化是放大器——放大紀律、也放大混亂，沒有那個底盤照抄只會更快崩壞
4. **速度上去、人審退場 → 累積認知債**：呼應 [[ai-testing-agile-quality#為什麼 GenAI 時代下，RD 更需要懂測試？（課程預告文，2026-07-02）|上方 METR 研究]]（資深工程師在熟悉專案上用 AI 反而慢 19%，自己卻以為快 20%）；junior 也失去從實作養判斷力的迴路。Ghostty 作者 Mitchell Hashimoto 反駁：「每日部署數」是空洞指標，該報使用者滿意度——部署 4,500 次跟 app 有沒有變好是兩件事

### 收斂

自動化負責驗證的「執行」，**判準**（什麼叫對、什麼叫夠好）仍是人的工作，且落在上游 SBE 那一段。Spotify 敢放手，是因為判準早就寫進測試與標準裡。agent 再強也無法替你定義品質——少了那塊，自動化只會讓你更快把沒驗證過的東西送進 production。

## 別再迷信自動化測試了（2026-07-04，LinkedIn 短文）

> 來源：敏捷三叔公 LinkedIn 貼文，同系列課程「不確定時代下的敏捷測試」(2026-07-26) 推廣文

短文版重申本筆記已有的核心論點，但補了兩個新角度：

### 新角度 1：全綠燈的擔心方向反過來了

- **以前**：擔心測試 cover 不夠多，沒經過修改的地方
- **現在**：AI 產出程式碼的速度比測試覆蓋率成長快好幾倍，一個下午能改一大片。全綠很可能只代表「測的都是沒被改到的地方」，覆蓋率追不上程式碼長大的速度

呼應上方 [[ai-testing-agile-quality#Spotify 案例：2,900 工程師、2,000 萬行 monorepo，verification 是核心瓶頸（2026-07-02）|Spotify 案例]]：品質指標「持平」不是因為驗證變好，是因為沒人注意到覆蓋率正在被拉開。

### 新角度 2：Review 本身就是一種測試

> 好好思考程式邏輯是否有不足，直接在來源處處理，這件事在 AI coding 時代反而更重要。

若每次都只看測試綠了就 merge，久了團隊沒人真的知道系統長什麼樣子，出事要查時不知從哪查起——這點跟 [[ai-testing-agile-quality#Spotify 案例：2,900 工程師、2,000 萬行 monorepo，verification 是核心瓶頸（2026-07-02）|Spotify 的「速度上去、人審退場 → 累積認知債」]]是同一件事的兩種說法。

### 具體化 Test Impact Analysis 的做法

上方表格（第 42 行）只列了名稱，這篇補了流程：分析這次程式碼變更 → 找出被影響的模組 → 只挑相關測試跑。初步分析可請 AI 做，但**最後測哪裡要有人看過再決定**——判準仍在人手上，這是本筆記反覆出現的收斂點。

## PM 不懂測試，AI 生的 Bug 就找上你（2026-07-08）

> 來源：敏捷三叔公，同系列課程推廣文

補上一個新視角：前面的內容多半對 RD/SM，這篇針對 **PM**——沒有專職 QA 的中小企業/新創，PM 往往就是最後一道防線。

### 四個情境

1. **AI 做出來的未必是你要的**：電商「結帳頁選運送地址」需求，AI 只改前端下拉選單，後端配送邏輯沒動，AI 自己寫的測試也只驗證「能不能選」。**規格沒講清楚的地方，AI 不會停下來問，會自己猜一個合理版本填進去**——需求越模糊，猜錯機率越高。呼應本筆記 [[ai-testing-agile-quality#AI 時代需重視的測試實踐|SBE 實踐]]：懂測試的 PM 會在需求階段就用具體範例把規格釘死。
2. **「測試全綠」不代表對**：AI 寫 code 又寫測試，驗證的是「程式碼實際做了什麼」，不是「使用者需要什麼」。這正是反覆出現的 **oracle problem**——AI 能生程式碼、生測試，但「正確的標準」得有人定，通常是最了解使用情境的 PM。
3. **Demo vibe coding 前先想「萬一」**：B2B 報表工具兩週做出來，投資人面前一查無資料就當機——AI 只在 happy path 上跑得順，開發過程沒人試過邊界情境。**vibe coding 的陷阱：做雛型的成本降到極低，但雛型與產品之間隔著一整片沒人走過的邊界**。
4. **沒有 QA，PM 就是最後防線**：AI 讓程式碼產量變三倍，驗證人力沒變，關鍵差異在 PM 能不能：用具體範例讓 AI 少猜、設計有效驗證路徑判斷產出可信度、模擬使用者最常犯的錯、區分「重大錯誤」與「尚可接受」。

### 收斂

> AI coding 時代真正稀缺的能力，是把需求講清楚（最好清楚到用範例表達），以及判斷做出來的東西對不對。前者決定 AI 猜錯的機率，後者決定錯的東西會不會流到使用者面前。

跟本筆記其他章節的核心命題完全一致，這篇只是換了 **PM 視角**再講一次：懂測試不代表要當 QA 或自己寫自動化測試，而是「講清楚需求」+「判斷對不對」這兩件事，本質都是測試思維。

## 抄來的最佳實踐，為什麼在你的團隊行不通？——用 Cynefin 看 AI 時代的測試導入（2026-07-08）

> 來源：敏捷三叔公，同系列課程推廣文

補上一個框架視角：前面各篇列了一串該上的實踐（SBE、Eval Gate、Self-Healing CI、Test Impact Analysis、Adversarial Testing、Audit Trail...），這篇回答「為什麼全部一起上反而會失敗」。

### Cynefin 三種領域的誤判

| 領域 | 特徵 | 錯誤動作 |
|------|------|---------|
| Clear（明顯） | 有標準答案 | 老闆常誤判成這層：「照最佳實踐做就好」 |
| Complicated（繁雜） | 找專家分析、規劃藍圖 | 團隊常誤判成這層：「抄大廠 playbook、一次到位」 |
| **Complex（複雜）** | **因果只能事後回溯，事前沒人知道哪個實踐有效** | **AI 測試導入的真實所在地** |

模型每季在變、團隊用 AI 的方式每月在變——別人的因果關係不會在你家重演。複雜領域的正確動作是 **probe–sense–respond**：小規模安全可失敗的實驗，看訊號，放大有效的、收掉無效的。

### 案例：阿哲的四個探測（把本筆記既有實踐串成一條時間軸）

| 探測 | 對應本筆記既有實踐 | 訊號 | 結果 |
|------|-------------------|------|------|
| ① 測試左移 | [[ai-testing-agile-quality#AI 時代需重視的測試實踐\|SBE]]：範例+需求一起餵給 AI | 正 | 放大，推廣到其他功能 |
| ② 測試左移 | AI Code Quality Gate（PR 自動擋壞味道） | 負（誤判率高，工程師繞過） | 先收掉，調規則後小範圍重上 |
| ③ 持續整合 | [[ai-testing-agile-quality#別再迷信自動化測試了（2026-07-04，LinkedIn 短文）\|Test Impact Analysis]] + Self-Healing CI，CI 從 90 分鐘縮到 20 分鐘 | 正 | 保留 |
| ④ 持續測試 | [[ai-testing-agile-quality#Eval Gate 深探：三層架構 + 雙信號線\|Eval Gate]] + Adversarial Testing（LLM 輸出非確定性，傳統斷言測不了） | 本身就住在複雜領域 | 沒有測完的一天，持續探測 |

最後一組探測放在生產環境：Continuous Verification + Audit Trail——呼應本筆記「測試右移」概念，**驗證不在部署那一刻結束，生產環境本身就是感知（sense）的一部分**。

### 收斂

半年後阿哲的團隊留下六個實踐、收掉一個、一個仍在觀察名單——這份清單跟大廠 playbook 長得不一樣，是用**自己的訊號**換來的，不是抄來的。

跟本筆記核心命題一致：AI 是放大器，但這篇補了方法論——**怎麼決定哪個實踐該留**。答案不是找更權威的 playbook，是縮小實驗範圍、跑起來、看訊號。

## 九個實踐該先衝哪裡？三波導入順序（2026-07-12）

> 來源：敏捷三叔公，同系列社群貼文

上面 Cynefin 那篇講的是「不要一次全上、要用探測」，這篇補上更具體的答案：**九個實踐照什麼順序做，才不會第一仗就陣亡**。

排序邏輯只有一條：**改動最小、訊號最快的先做**。用最少投資換到「AI 產出確實有問題、而且我們抓得到」的證據，再拿證據去談需要組織配合的大工程。

### 開始前先盤點三件事

- **CI 現況**：每次 PR 會不會自動跑？沒有 CI 的話，先把 CI 立起來
- **測試存量**：自動化測試蓋到哪，決定 Test Impact Analysis 什麼時候能上
- **AI 使用紀錄**：prompt 和產出有沒有留痕跡？通常沒有，這就是要補的洞

### ⚡ 第一波：一到兩週看到訊號

| 實踐 | 為什麼先做 | 工具 | 啟動成本 |
|------|-----------|------|---------|
| **AI Code Quality Gate** | CI 通常已有 linter，只是加規則；每次 PR 產生具體數字（擋下幾筆重複碼、幾個過度防禦 try-catch），是說服團隊的彈藥 | ESLint/Ruff 收緊規則、jscpd 抓重複、Semgrep 自訂規則、SonarQube 調門檻 | 1~2 天，熟 CI 的工程師就能做 |
| **Adversarial Testing** | 開第二個 session 或換模型打自己的 code，找邊界條件；不用改流程，常第一次就抓到 bug，是最好的內部宣傳 | Claude Code 開第二 session 當攻擊方，進階可配 Hypothesis(Python)/fast-check(JS) | 半天~1天，一人一模組就能開始 |
| **Audit Trail** | 技術門檻幾乎零，但是後面所有實踐的資料地基——沒紀錄，Eval Gate 連 dataset 都湊不出來。金融/醫療等法遵產業應升格第一優先 | PR template 加欄位(AI參與程度/prompt摘要)、commit慣例、session log歸檔 | 1天以內，主要是養成習慣 |

### 🤝 第二波：一到兩個月，需要跨角色配合

| 實踐 | 為什麼放這裡 | 工具 | 啟動成本 |
|------|-------------|------|---------|
| **Specification by Example** | 價值最高但卡組織面——要PO/開發/測試坐下來談例子，需要信任和時間；太早推容易被「大家很忙」磨死，等第一波數據證明驗證是瓶頸再談就順了 | Example Mapping 工作坊(白板+便利貼)，落地成規格再選 Cucumber/Reqnroll/Gauge | 每個feature 30~60分鐘對話，真正成本是PO願意到場 |
| **Test Impact Analysis** | AI讓改動量暴增後，全量跑測試的回饋時間變成新瓶頸，這時才有感；前提是要有一定測試存量 | JVM: Gradle/Maven增量測試外掛；前端: Jest changedSince；細緻可用coverage資料自建mapping | 3~5天，看codebase複雜度 |
| **Secure-by-Default** | AI產出常帶過時/不安全預設寫法，與其事後掃，不如先鋪好安全鷹架和範本，讓AI一開始就踩在對的地基上 | 內部project template、Dependabot/Renovate、Trivy+Semgrep安全規則進CI | 1~2週建範本，之後持續維護 |

### 🏗 第三波：一季以上，吃前面累積的資料和harness

- **Eval Gate**：需要 dataset，從第一波 Audit Trail 累積的真實案例來——沒有紀錄就是空中樓閣。promptfoo可起步，多數團隊最後自建harness，成本以週計，還要專職維護eval集
- **Self-Healing CI**：讓agent自動修紅燈很美，但前提是CI訊號要可信——flaky test一堆的團隊先修flaky，不然是讓AI在爛地基上蓋違建，成本以月計
- **Continuous Verification**：把驗證延伸到production：Pact contract testing、feature flag配canary、OpenTelemetry接監控。這是終局不是起點，成本以季計

### 三波總結

> ⚡ 第一波用幾天成本換「證據」
> 🤝 第二波拿證據換「組織配合」
> 🏗 第三波拿資料和harness蓋「大工程」
>
> 順序反過來走，大概率死在第一仗。

跟本筆記其他章節呼應：這篇把「阿哲的四個探測」案例背後的邏輯講得更白——為什麼 SBE 排在 Audit Trail 之後（組織信任要先靠數據換）、為什麼 Eval Gate 排在最後（要吃 Audit Trail 的資料）。是同一套 Cynefin probe-sense-respond 思維，換成具體的「先做什麼」清單。
