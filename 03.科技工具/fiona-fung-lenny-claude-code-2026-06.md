# Fiona Fung × Lenny：寫程式不再是瓶頸之後，瓶頸換去哪了

> **Fiona Fung** — Anthropic Claude Code & Claude Cowork 產品暨工程負責人
> 25 年工程資歷，建立 Facebook Marketplace（年 GMV $1000 億），Meta 管過 500 人

---

## 核心數字

**Anthropic 工程師每季 code 產出 = 2021-2025 年間的 8 倍**

不只是更多人提交，而是不同職能的人也在提交——設計師、PM、Claude Code 團隊所有人都在 check in code。

---

## 1. 瓶頸換去哪了？

| 年代 | 瓶頸 | 原因 |
|------|------|------|
| 以前 | **寫程式**（工程時間） | 軟體燒光碟上架，截止日期硬，有限時間極大化產出 |
| 現在 | **驗證與品質** | 產出量暴增 8 倍，產出者角色也更多元 |

寫程式不再是瓶頸。典範轉移了。

---

## 2. 2026 AI 化軟體團隊長什麼樣

**最大轉變：角色界線正在融合，每個人都在往 builder 靠攏**

Fiona 的具體做法：
- 開一個常駐的 **Claude Code remote session** 掛進團隊所有 repo
- 同時存取所有 Slack 頻道 + 追蹤指標
- 每月跟團隊做 session 回顧：重點項目、出了哪些產品、市場反應、回饋訊號
- 用途從「產生 PR / 修 bug」擴大到「支持管理對話」、「歸納跨團隊 incident 主題」、「找品質缺口」

> 以前整個過程太手動了。八倍的產出量，這幾乎是唯一能跟上的方式。

---

## 3. 晨間管理 Routine

**過去**：泡咖啡 → 瀏覽回饋管道 → 看能幫什麼 → 想下一步

**現在（Claude Routines）**：
- Routine 每天固定時間自動執行
- 自動掃描所有回饋管道（Slack、Email、社群媒體、LinkedIn）
- 歸納主題 → 找品質缺口 → 產生可直接 review 的 PR

**醒來就有決策材料，不是資訊，是已整理好的產物**

作為管理者，把日常要做的事寫成 routine：「專案進度如何？誰需要支援？有哪些 polish 可以改？」Claude 執行後呈現結果讓你 review，甚至到一定程度可以給更多自主權直接執行。

---

## 4. Code Review 的演進

舊做法：人工 reviewer → 大瓶頸

**新做法：Spec-Driven Review**

1. 把「什麼叫做好」的框架寫成 spec
2. 把 spec check in 到 repo（確保 spec 跟 code 同步更新）
3. Claude Code Review 驗證 code 是否符合 spec

> 這是 TDD 的進化版：過去寫測試是「先吃花椰菜」——原則對但很痛苦；現在「寫測試的稅」被自動化了。

Fiona 自己修第一個 bug 的流程：Claude 先寫測試 → 確認測試失敗 → Claude 修復 → 測試通過。

---

## 5. Bad vs Sad 品質框架

| 類型 | 定義 | 例子（CLI） |
|------|------|------------|
| **Bad** | 嚴重、不可恢復 | crash（工作全部遺失） |
| **Sad** | 痛點但可恢復 | 畫面閃爍（體驗差但不影響工作） |

注意：累積太多 sad 可能堆疊成 bad。

每個團隊按自己負責的產品面定義 bad/sad，用統一框架俯瞰整體體驗品質，而不是盯著一堆不同的效能數字。

**品質保衛的核心**：投資閉環系統，讓 agent 自己知道成功的定義，在框架內自行解決，而不是花更多時間人工 review（速度根本跟不上）。

---

## 6. 招募：只找兩種人

| 類型 | 特質 |
|------|------|
| **創意 Builder** | 有產品 sense，全端思維，能端到端擁有產品體驗 |
| **深度系統專家** | 有系統底蘊，負責困難核心（trust but verify：這些領域仍需深度） |

Fiona 剛加入 Claude Code 時，發現有好的產品通才，但缺少分散式系統背景的人。

---

## 7. 野心重新定義

10X 工程師的思維轉變：
- 以前聽到功能想法 → 「不行，太難太複雜了」
- 現在 → 「完全可以，讓 Claude Code 去做就好了」

一個不是 mobile 工程師的人，因為有了 Claude，可以說「我現在也能做 Android 開發了。」

**AI 拉高了每個人能做到的天花板。現在問題是你的野心能有多大。**

---

## 8. 成長心態 + 面對恐懼

**最擁抱 AI 的工程師，差距最大的特質：**
1. **成長心態**（過去讓你成功的方式，不代表以後還適用）
2. **面對恐懼的方式**——不被未知嚇退，主動迎上

遇到恐懼時：問自己「有什麼是我能做的？什麼在我的掌控範圍內？」而不是「這都在我掌控之外」。

Fiona 最愛的兩句話：
- 「如果你不害怕，你會做什麼？」
- 「偶爾做一件讓你害怕的事。」

> Lenny 引用 Joseph Campbell：「你害怕進入的那個洞穴，藏著你追尋的寶藏。」

---

## 9. Token Maxing vs ROI

> **「別把動作誤認為進步。」（Don't forsake motion for progress.）**

- 衡量標準：不看 token 用多少，看動作有沒有變成成果
- Token maxing = 程式碼行數迷思 2.0，衡量的是行動而不是影響力
- 生產力指標需要隨環境演進，過去有意義的指標不代表現在還適用

**Facebook Marketplace 教訓**：用「賣家數量」當擴展門檻，差點錯過一個因為有 power sellers 而運作良好的區域。後來把 power sellers 因素納入指標。

---

## 10. 非同步工作的新挑戰

Claude Routines 讓工作從同步 prompt 進化到非同步 agent 編排：
- 以前：下 prompt → 等結果
- 現在：用 routine 產生 prompts → 啟動多個 agent → 醒來 review 結果

**新問題**：情境切換負荷增加。Fiona 有一次啟動太多非同步 agent 才意識到這個問題。以前要留下專注時間寫程式，現在反而要留下專注時間追趕所有非同步工作的進度。

**好消息**：agent 能幫你重新進入狀態（「這是你之前在做的」），切換的重啟成本降低了。**這件事同時變好也變壞了。**

---

## 11. 所有管理者先從 IC 做起

新主管進來一律先當 IC（個人貢獻者），再開始管人。原因：

- 先花時間理解產品、熟悉 codebase，再負責支持人
- 如果一來就打開「管理者工具箱」，會錯過跟團隊建立默契的機會
- 先當隊友、給出使用者回饋，建立的信任感完全不同

Fiona 自己在 Meta 也是如此，面試進去是管理職但第一季先當 IC。

> 「她有很多當了多年管理者的朋友，因為 Claude Code 重新開始寫程式了。」

## 12. 技能退化：Double-Click on Your Dependencies

**「雙重點擊你的依賴」（Double-click on your dependencies）**

- 理解你做的每個改動背後的架構
- 因為當依賴改變時，你才能意識到並利用新的能力
- 也許有一天模型強到不需要了，但以目前速度，這個能力仍然重要

Boris 當初親手寫了早期程式碼，雖然不再這樣做了，但那些知識是無法取代的。

## 13. Dogfooding 的執念

> 「作為領導者，如果你只看指標和儀表板或簡報，你會跟產品失去觸感。」

Fiona 每個產品的第一件事：親自使用。

具體案例：
- **Marketplace**：把舊 MacBook Air 掛上賣，馬上遇到新型態詐騙（才發現新攻擊手法）
- **智利**：帶三人小組去，拿 Android 手機測，發現 LTE 比美國慢，Marketplace 根本載不出來 → 巨大成長阻礙，不親自去根本不知道
- **Fiona 的家**：VR/AR 時代她的家裡環境神奇地能重現特殊 bug（地板高度問題）

> **Jeff Bezos**：「如果你有數據也有軼事，相信軼事而不是數據。」Fiona 非常認同。

**給不能親自使用自己產品的人**：去見客戶。每次客戶拜訪都會學到新東西。

---

## 14. Loneliness：配對程式設計午餐

Claude Code 團隊發現問題：大家都跟自己的 agent 一起工作，**開始覺得孤獨**。

解決方案：
- 配對程式設計午餐（pair-wise programming lunch）：發現每個人使用 Claude Code 的方式都不同，配對工作能互相學習
- Hackathon：確保團隊互動

> 就像小孩子的「平行遊戲」（parallel play）——你在旁邊做自己的事，但光是看別人怎麼做就很有收穫。

---

## 15. JIT 月度規劃

Fiona 剛加入時帶進六個月路線圖做法，三個月後發現大家根本沒有再回去看，因為變化太快。

**現在：JIT Planning（Just-in-Time）**
- 以一個月為週期
- 一個簡單試算表列出本月最高優先事項
- 每週快速 check in：「這些還是本月優先事項嗎？」
- 每六個月把整個團隊聚在一起討論主題方向（不做細部規劃）

**建議所有人做的事**：明確允許砍掉不再有用的流程。挑一個你害怕做的、很費力的、太手動的流程，先問它是不是還有用。

---

## 16. 潛在需求（Latent Demand）

Anthropic 為什麼總能早一步發現大機會？

**持續關注 latent demand + 自己是第一批客戶**

- Cowork 的誕生：注意到很多非工程師也在用 Claude Code → 「能不能把這個體驗做得更好？」
- 永遠保持傾聽回饋、持續迭代，同時留意那些「你沒預期到的使用場景」
- 當你看到人們費盡心思讓某個功能運作，那就是把體驗變順暢的機會

---

## 17. 工程的未來：教育與學徒制

Fiona 的擔憂：如何培養下一代工程師？

- 過去：暑期實習做小專案，慢慢積累經驗
- 現在：這條路變了，怎麼快轉那些工程素養？
- 也許走向 fellowship 或學徒制（apprenticeship）

**值得定期重新嘗試曾經失敗的事情**，因為模型進步速度是指數級的，上一個版本不行，下一個版本可能就行了。

> Lenny：「去做那些幾乎快要可以運作的東西，因為一旦模型到了那個水準，你就會遠遠領先所有人。」

---

## 18. 管理文化

最讓 Fiona 睡不著的事：**團隊文化**

- 文化是活的，展現在我們怎麼對待彼此、怎麼互相支持
- One team 心態：「當你快到終點線的時候回頭看，有沒有人需要你幫忙，因為我們是一起抵達終點的。」
- 明確要求每個人分享「什麼沒有進展順利」，不只講好事
- 最怕的：問「一切還好嗎？」聽到「一切都很好」→ 「完蛋了，我做得不夠好」

> 「犯新的錯」：如果目標是零失誤，代表走得不夠快或太過謹慎。

## 19. PM 和 Data Science 角色的轉變

**PM**：不再因等待工程資源而卡住，有想法可以自己動手做 → 驗證更重要

**Data Scientist**：工作從「自己做分析」變成「review AI 做的（可能有問題的）分析」

Fiona 認為設計和 data science 是下一波可以大幅改善體驗的領域。

## 閃電問答

**推薦書**：
- Margaret Atwood（推想小說，「這真的可能發生在某個社會」）
