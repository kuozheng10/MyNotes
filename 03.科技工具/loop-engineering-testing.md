## 核心轉變

| 過去 | Loop Engineering |
|------|-----------------|
| 自己打 prompt → 等 AI → 再打下一句 | 設計一個「會自己下指令」的系統 |
| 測試 = 寫完 code 後跑的安全網 | 測試 = **決定 agent 何時收手的尺** |
| 人工翻一整排紅燈 | 早上開收件匣，只處理真正需要人腦的那條 |

## 四個實作元件

### 1. 測試當停止條件（/goal）

把可驗證的測試條件設為 loop 的 goal：
```
booking/ 所有測試通過 + booking-rules 全綠 + lint 乾淨 → 收手
```

**關鍵推論**：迴圈停在哪裡，完全看你的測試畫到哪裡。
- 那條「商務車廂只剩 2 張卻要訂 3 張」的情境沒寫成測試 → loop 不知道這是錯的 → 超賣版本被判定完工
- **SBE/BDD 範例寫得多完整，直接決定這個迴圈能不能信**

### 2. 獨立驗收 Agent（Adversarial Review）

- 寫的 agent 自己驗收 → 太寬容（剛說服自己這樣寫是對的）
- 另養一隻 **verifier agent**，指令：「拿訂票規格 + 現有測試，去挑剛產出程式碼的毛病，特別注意併發訂票、剩餘張數邊界、付款失敗回滾」
- 它的工作就是當壞人，找「測試沒蓋到但規格有講」的縫
- **你敢放手走開的唯一理由**：有一隻信得過的驗收 agent

### 3. 排程巡檢（Automation）

每天上班前自動跑：
1. 讀昨天 CI 失敗
2. 讀最近改過的 commit
3. 讀還開著的 bug 票
4. 整理清單 → 能修的開 worktree 派 agent 修 → 搞不定的丟收件匣

**Worktree 重點**：讓兩隻 agent 同時改 code 不會互相干擾

### 4. 持久記憶（Memory）

Agent 每跑完一輪就忘光 → 需要「活在對話之外」的記憶

記憶檔放在 repo（markdown 或 Linear 看板）要記錄：
- 哪些情境試過了
- 哪些過了、哪些還開著
- 哪一條昨天修到一半

→ 明天那一輪才接得回今天停的地方

## 一天的樣子（訂票範例）

```
08:00 排程醒來
  → 讀 CI 失敗 + commit + issue
  → 發現「商務車廂剩餘張數」兩條測試紅燈
  → 開 worktree，派 agent 修
  → verifier 對規格驗收
  → 一條過 → 開 PR 連票
  → 一條牽涉金流回滾 → 寫進記憶檔 → 丟收件匣

09:00 你進公司
  → 打開收件匣，只處理那一條需要人腦的
```

你只「設計」了這個流程一次，中間沒有一步是你親手下指令的。

## 幾句老實話

**驗收終究是你的事**
- Loop 說「全綠」= 「我寫的那些測試都過了」
- 不保證程式真的沒問題

**理解債的隱性代價**
- 迴圈改 code 越快 → 你對那塊 code 越不熟（除非真的去讀）
- 同一個迴圈：有人用來在熟悉的模組上跑更快 vs. 有人用來逃避搞懂金流
- 迴圈分不出差別，**你分得出來**

## 起手建議

1. 先挑你最有把握、測試最完整的模組（不是金流！）
2. 寫清楚的 SKILL.md（專案說明）
3. 設一個綁著現有測試的 /goal
4. 加一隻 verifier agent
5. 最後才掛排程

**把測試養到夠格當那把尺，後面要不要往更自動的迴圈走，都還是你說了算。**
