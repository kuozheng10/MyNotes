# 導入 Scrum + AI Coding，為什麼還是卡？五個失效故事

> AI 把寫 code 的速度拉快，但五個老問題不但沒消失，反而被放得更大。

---

## 核心命題

AI coding 進場之後，真正稀缺的不再是「會寫程式的人」，而是：

1. **把需求講清楚的能力**（上游）
2. **把產出驗證到位的能力**（下游）

換了工具、換了儀式，腦袋沒換，就是換了形式。

---

## 五個失效模式

### 1. 只是換了儀式，沒換思維

**症狀：** 站會照開、Sprint Review 照做，主管把站會當點名，AI 之後變成「你昨天用 AI 產了多少 code」。

**AI 放大效果：** 衡量成功的方式沒換，更快做出沒人要的東西。

**根因：** 導入 Scrum 的動作完成了，但「為什麼要這樣做」沒有被內化。

---

### 2. PO 缺位，需求永遠模糊

**症狀：** PO 一週露面一次，需求文件丟著就消失。工程師憑經驗亂猜。

**AI 放大效果：** 以前猜錯頂多做錯一個功能，現在把模糊需求丟給 AI，半天生出一大包看起來完整的 code，跑得動、測得過，客戶一看：「這不是我要的。」重工從「一個功能」→「一整包」。

**根因：** AI 只能加速執行，無法取代「把需求講清楚」這件事，只有 PO 能做。

> 懷念 waterfall 時代規格書，其實懷念的是當年有人把需求講清楚。

---

### 3. 團隊還是各做各的

**症狀：** 跨職能只是形式，前端做前端、後端做後端。AI 讓「各做各的」更有藉口（卡關直接問 AI，比問隊友快）。

**AI 放大效果：** 每個人配 AI 高速產出，等到整合才發現介面對不上、資料格式打架。Sprint 結束只 demo 得出零碎功能，個人產能上去，團隊整體產出更難收斂。

**根因：** AI 提升個人生產力，但溝通與對齊沒有 AI 能幫忙做。

---

### 4. Scrum Master 盯錯地方

**症狀：** SM 被當秘書，或反過來變微經理。看板卡片動得漂亮、sprint burndown 很好看，就以為沒問題。

**AI 放大效果：** Code 產得更快，但有沒有被好好驗證、需求有沒有被理解對，SM 沒在追。Sprint Review 才發現跑得快不代表做得對。

**根因：** SM 真正該幫團隊盯的是「驗證跟不上產出的缺口」，不是「卡片移動速度」。

---

### 5. Sprint 變成迷你 waterfall

**症狀：** Sprint 開頭要求完整設計文件，開發集中在最後兩天，QA 最後一天意思意思測。

**AI 放大效果：** 以前工程師熬夜手寫，現在最後兩天叫 AI 把整包 code 生出來。產出速度看起來史無前例地快，可交付、可驗證兩件事一樣都沒做到。兩個月後進度條動了，真正能交付的東西沒動。

**根因：** 分了 Sprint 但沒有改變「先全設計、再全開發、最後一起測」的思維。

---

## 對照表

| 失效模式 | 傳統痛點 | AI 之後的放大版 |
|---------|---------|----------------|
| 儀式沒換思維 | 站會變點名 | 量 code 產出量取代量交付價值 |
| PO 缺位 | 猜錯一個功能 | 猜錯一整包，全部重工 |
| 各做各的 | 整合有點卡 | 個人飛速 + 團隊零碎 demo |
| SM 盯錯地方 | 行政事項 | 驗證缺口被忽視 |
| 迷你 waterfall | 最後衝刺熬夜 | AI 生整包但無法交付 |

## 從燃盡圖看 AI Coding 時代的開發問題（2026-07-04 補充）

> 來源：敏捷三叔公，同系列課程預告文

### Scrum = PDCA，但 AI 時代重心從 Do 轉移到 Check

Scrum 用 sprint（Plan-Do-Check-Act）拆解不確定性與複雜度。AI coding 時代多了第三個問題：**AI 產出到底對不對**。「生得出來」跟「是對的」是兩回事。過去 PDCA 裡 Do 最花時間，現在 **Do 變得很便宜，Check 變成瓶頸**。

### Burndown 斷崖 = 藏起來的 waterfall

Burndown chart 常見「最後一刻才往下掉」的斷崖形狀。AI coding 讓斷崖更陡：AI 讓「產出」變快，task 看起來一直在動、一直有進度，但真正整合、驗證、讓 story 真的 done 全部擠在最後——**燒的是 story point，累積的是還沒驗證過的程式碼**。

這正好呼應本筆記上方 **[[scrum-ai-five-fails-2026-06#5. Sprint 變成迷你 waterfall|失效模式 5：Sprint 變成迷你 waterfall]]**——同一個病徵，這篇補上了圖像化的診斷工具（燃盡圖斷崖）。

### Little's Law：WIP 是你唯一能控制的領先指標

```
avg cycle time = avg WIP / avg throughput
```

cycle time、throughput 是落後指標，真正能控制的是 **WIP 數量**（領先指標）。AI coding 時代 WIP 爆炸風險特別高：開一個新工作幾乎零成本（開對話、丟 prompt），可以同時讓好幾個 agent 平行跑好幾個功能——但每個都只做一半、都沒驗證整合，cycle time 反而被拉長。

**結論：越是能用 AI 平行處理，越要刻意壓低 WIP 數字。**

### 交付節奏可以跟 Sprint 節奏脫鉤

不用等到迭代最後才交付、才 demo。完成一個 story 就可以馬上交付、馬上拿到回饋。AI 時代這件事價值更高：AI 能快速產出一堆東西，但**回饋（AI 做的方向對不對）才是最稀缺的資源**。越早做完一個、驗證一個、交付一個，就越早拿到回饋去修正 AI 接下來的方向，整批做錯的機會就越小。
