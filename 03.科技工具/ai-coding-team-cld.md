# AI 把 WIP 的天然煞車拆了 — AI Coding 時代的團隊 CLD

> 來源：Bas Vodde 演講概念延伸，課程「不確定時代下的大規模敏捷 – LeSS 入門篇」
> 儲存日期：2026-06-11
> 標籤：#agile #ai-coding #wip #cld #less #team

---

## 核心命題

Bas Vodde 的原始 CLD（因果迴路圖）描述「多人圍著同一個項目工作」的飛輪。  
AI coding 時代有五條因果關係改變了，最危險的一條：**WIP 的天然煞車消失**。

---

## 原圖推論鏈

```
多人投入同一項目
  → 團隊同時在做的項目數（WIP）下降
  → 大家一起工作的時間增加
  → 快速反應改變、學習量上升
```

底層撐起這一切：TDD、自動化測試、重構、程式碼共有、任務切小、頻繁整合。

---

## 五個在 AI 時代改變的地方

### 1. WIP 天然煞車消失（最危險）

- 過去：WIP 上限 = 人手數量（自然限制）
- 現在：一人可同時開 3-5 個 agent，個人平行度直接墊高整體 WIP
- 新增因果箭頭：「每人同時監督的 agent 數量」→ 推高團隊 WIP
- **WIP 管理從自然發生 → 必須刻意設計**

### 2. 一起工作的成員擴大到 agent

- Mob programming 的參與者：人 + agent
- 模式：一人下指令 → agent 動手寫 → 其他人盯產出即時討論
- **Agent 給不了的：學習量**（學習來自人一起檢視、討論、做決策）

### 3. 理解債（Comprehension Debt）進入因果鏈

新增兩個變數：
- 「未經人檢視的 AI 產出量」
- 「團隊對程式碼的理解程度」

因果鏈：
```
未檢視產出↑ → 理解程度↓ → 改變代價↑
```

一起工作的時間在 AI 時代多了一個作用：**壓低未經人檢視的產出量**。

### 4. 信心的地基換了

- 原圖：TDD → 自動化測試 → 修改的信心
- 新圖：信心來自整套 **Validation Harness**（自動化測試 + eval gate + AI code review）
- Harness 有效的前提：**Sensors 要知道「對」長什麼樣子** → SBE（規格與驗收範例）清晰度

> 範例寫不清楚，harness 只能查程式有沒有跑，查不了做的對不對。

### 5. 程式碼共有 → 程式碼與 Context 共有

Collective ownership 擴大範圍：
- 原：程式碼共有
- 新：程式碼 + **CLAUDE.md / 規格範例 / prompt / skill** 都是團隊共同資產

Context 不共有 → 每人 agent 產出風格各異 → 整合成本↑ → 多人投入同一項目的能力↓

---

## 健康度指標（每週看兩個數字）

| 指標 | 警示訊號 |
|------|----------|
| 每人平均同時開幾個 agent | 越高越危險 |
| 團隊同時在做幾個項目（WIP） | 兩個數字一起往上走 = 第一條新箭頭在發作 |

**最先反映在**：沒有人 review 過就合併的 PR 比例。

---

## 對派哥工作的啟示

- 用多個 agent 並行時要刻意設 WIP 上限（per session）
- CLAUDE.md / skill / SBE 是 context 共有的基礎設施，值得持續投資
- PR review 不能跳過，即使 agent 產出看起來正確
