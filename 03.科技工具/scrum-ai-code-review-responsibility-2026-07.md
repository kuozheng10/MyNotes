# 「那是 AI 產的，不關我的事」——當責任從人與人之間，退到人與 AI 之間

> 一面照妖鏡：想知道一個團隊 AI 時代的 Scrum 做得好不好，不要看產出速度翻了幾倍，去看他們 Sprint Planning 怎麼討論、怎麼互相面對。

---

## 核心命題

AI coding 時代最危險的一句話：**「那是 AI 產的，出事不是我的問題。」**

責任的邊界，從「人與人之間」（這不是我負責的模組）退化成「人與 AI 之間」（這是他的 AI 產的，我沒看）。Scrum 的自組織精神講的是「團隊共同為產出負責」，但當每個人只跟自己的 AI 結對、只顧自己的 prompt，團隊就從「一個團隊」退化成「一群配了 AI 的個體戶」。

---

## 案例：飛翔貓科技的故事

### 起點：看起來「進化了」的團隊

飛翔貓科技的工程團隊全面導入 Cursor、Claude Code、Copilot，Sprint Planning 開得比以前更順、估點更快——因為每個任務的回答都一樣：「這個丟給 AI 很快啦。」

Scrum Master 阿沁卻觀察到五個不對勁的現象：

| 現象 | 具體樣貌 |
|------|---------|
| 任務切割更細更個人化 | 一人配一個 AI，各做各的；不再討論怎麼拆功能，反正 AI 產得快，一人包一塊最有效率 |
| Code review 名存實亡 | PR 越來越大越來越多，大家心照不宣按 Approve：「那是他的 AI 產的 code，我看不完，也不關我的事」 |
| 團隊不再討論架構設計 | 討論重心從「這個功能該怎麼做」變成「這個 prompt 要怎麼下」——而 prompt 是個人的，不是團隊的 |
| 沒有人 pairing 了 | 每個人都跟自己的 AI 結對，產出表面翻倍，人與人之間的知識流動歸零 |
| 新人成長停滯 | 新人小傑負責的功能 AI 都能產出、都能動，但問「為什麼這樣設計」答不出來——沒有資深工程師發現，因為沒有人看過他的 code |

這樣的 Sprint Planning，本質是把事情「分配給每個人的 AI」，不是團隊「一起決定怎麼前進」。

### 危機：權限模組出包

某個 Sprint 的核心功能是帳號權限模組。資深工程師阿邦用 AI 兩天產出幾千行 code，測試全綠，順利上線。大家都說：「看吧，AI 時代就是快。」

三週後，客戶回報：**某些情況下，一般使用者可以看到管理員的資料。**

而阿邦正好生病請假。團隊打開那幾千行 AI 產的 code，沒人看得懂設計脈絡——測試是 AI 產的，涵蓋了「AI 認為該測的情境」，卻剛好漏掉這個權限邊界。沒有人知道當初的 prompt 是什麼、做過哪些取捨、哪些地方阿邦自己也沒仔細看過。

> 以前是「只有阿邦懂這個模組」；現在是「連阿邦都不完全懂這個模組，而其他人連碰都沒碰過」。

Sprint Review 時 PO 的一句話讓全場沉默：

> 「我們的產出速度變成三倍，但為什麼我覺得，我們對自己系統的掌握度變成三分之一？」

### 攤牌：「你還要我 review 別人的？」

阿沁想改變規則：重要功能不再一人包辦，AI 產出的 code 必須有第二個人真正看懂並簽名負責，關鍵模組改成兩人一起設計、一起下 prompt、一起驗證。

工程師小磊直接開砲：

> 「為什麼我要看別人 AI 產的 code？我自己 AI 產的都看不完了。現在 AI 產 code 的速度是我們看 code 速度的十倍，你還要我 review 別人的？這樣搞一搞，最後只是變相讓我們多做事情。你說這是品質，但我聽起來就是把 AI 省下來的時間，全部拿去補 AI 挖的洞。」

沒人接話，但明顯有人在心裡默默點頭。文章點出這不是個人問題，是 AI coding 時代團隊的共同掙扎，背後其實是三種不同心理：

1. **真的累**——AI 讓寫 code 變快，但驗證量暴增，PR 像潮水一樣湧來看不完
2. **不敢承認**——自己其實也沒完全看懂 AI 幫自己產的東西
3. **習慣了新邊界感**——「我負責我的 prompt，你負責你的 prompt」

以前的藉口是「這不是我負責的模組」，現在的藉口是「這是 AI 產的，出事也不是我的問題」。

### 轉折：不說教，讓團隊自己面對

阿沁沒有反駁也沒急著說服，只丟出一個問題：

> 「你們希望待在什麼樣的團隊？是一個『每個人都很快，但沒有人知道系統長什麼樣子』的團隊，還是一個『AI 幫我們加速，但我們仍然共同理解、共同負責』的團隊？」

接著設計了一個匿名活動「我想要／不想要的團隊」，貼在牆上的紙條包括：

- 「我不想要哪天出事時，發現全公司沒有一個『人』看得懂那段 code」
- 「我想要 AI 幫我變強，而不是幫我變得可以不用思考」
- 「我不想再假裝我 review 過那些 PR 了」
- 「我不想因為誰生病，就發現我們其實只是一群坐在一起的個體戶」

看完之後小磊鬆口：

> 「好啦…我也不是不願意看。老實說，上次那個權限的 bug，我心裡有底——因為我自己的 code 我也沒有每一段都看懂。只是大家都在衝速度，我停下來看，就好像是我在拖累團隊。」

阿沁沒有要求一次改掉所有事，只提出一個最小可行的實驗：

> 「這個 Sprint 我們只試一件事——挑一個最關鍵的功能，你和阿邦一起下 prompt、一起 review AI 的產出，一天就好。不是幫他做事，是讓這個模組從『一個人加一個 AI 懂』，變成『兩個人都懂』。覺得沒效就停，可以嗎？」

那個 Sprint，小磊第一次在別人的 AI 產出裡，抓到了一個測試沒涵蓋到的邊界條件。

## 提煉出的原則

**Scrum 是自組織，不是「各自外包給 AI」。** AI coding 時代團隊要問自己四個問題：

1. 是否願意共同為 AI 的產出負責，而不是把責任推給工具？
2. 是否有心理安全，讓人可以說出：「這段 AI 產的 code，其實我沒完全看懂」？
3. 是否把「驗證」和「理解」當成團隊的日常，而不是速度的絆腳石？
4. Sprint Planning 時，是不是只在把任務分給每個人的 AI，而不是設計一個「人與人仍然互相學習」的節奏？

> AI 可以外包你的打字，但不能外包你的責任；可以加速你的產出，但不能代替你們成為一個團隊。

判斷一個團隊 AI 時代 Scrum 做得好不好，不要看產出速度翻了幾倍，去看 Sprint Planning 時他們怎麼討論、怎麼選擇、怎麼互相面對——那裡才看得出，這個團隊究竟只是「一群配了 AI 的個人」，還是「真的成為了一個團隊」。

## 怎麼解決 code review（派哥問答，2026-07-11）

> 文章點出問題（AI 產出速度追不上 review 速度），但沒給具體解法。以下是針對「怎解決 code review」這個問題的實際建議。

**核心建議：review 的重點從「逐行讀 code」換成「風險分級 + 逼你證明看懂」。**

逐行讀已經追不上 AI 的產出速度，硬要求全 review 只會逼大家假裝按 approve（就是小磊崩潰的那件事——「這樣搞只是變相多做事」）。實際做法拆三層：

1. **高風險模組才強制配對** — 權限、金流、資料外洩相關的模組，才要求兩人一起下 prompt、一起 review（就是阿沁後來實際做的：不是全部都配對，只挑最會出事的）
2. **其他模組不用別人重讀，但作者要能講出設計理由** — 「這段為什麼這樣設計、AI 幫我做了什麼取捨」講不出來，就代表自己都沒看懂，不能合併。這比找第二個人重讀更省力，也更能逼出真的理解
3. **力氣從「讀 code」轉成「補測試」** — 故事裡漏掉的是「權限邊界」這種 AI 自己想不到要測的情境。與其人工重讀幾千行，不如要求關鍵功能一定要有人（不是 AI）主動想過「這裡最容易出包的邊界是什麼」，再叫 AI 去補對應的測試

**取捨**：這樣做會漏掉「低風險模組裡的意外雷」，但換來團隊真的看得完該看的東西。全面要求逐行 review，代價是大家用「假裝看過」應付——那更危險，等於自欺欺人的安全感。

**個人適用版（一人團隊，例如派哥 + Claude Code）**：不用每次都要求完整重講邏輯，但碰到權限/金額/會動到真錢的地方（例如 2026-07-10 那次國泰信用卡 OCR 漏記交易的 bug），值得停下來多問一句「這樣改會不會漏掉什麼」——本質上跟上面第 1 點「風險分級」是同一件事，差別只是配對的對象從「另一個工程師」換成「多問 AI 一輪」。
