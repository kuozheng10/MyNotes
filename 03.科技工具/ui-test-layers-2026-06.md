# UI 自動化測試與分層測試的關係

> **核心觀念**：不要把所有驗證都塞進 UI / E2E Test。
> 每一層測試負責不同的驗證問題，才能讓測試策略可維護、成本合理。

---

## 三層測試的責任邊界

| 層次 | 驗證的問題 | 特性 |
|------|-----------|------|
| **Unit Test** | 規則本身是否正確 | 執行快、定位精準、無瀏覽器 |
| **Integration Test** | 規則如何接入系統流程 | 跨模組整合、確認資料流 |
| **UI / E2E Test** | 使用者能否在畫面上完成流程 | 最接近使用者視角、成本最高 |

---

## 各層詳細說明

### 🔹 Unit Test

**驗證對象**：單一函式、類別、模組、純邏輯。

- 金額計算、條件判斷、資料轉換
- 不需要啟動瀏覽器或呼叫外部服務
- 執行快、失敗原因容易定位
- 適合驗證「規則本身有沒有算錯」

### 🔹 Integration Test

**驗證對象**：多個模組之間的整合結果。

- 後端 API、資料存取、服務協作
- 確認資料在系統內部正確流動
- 可以不涵蓋完整畫面操作
- 適合驗證「各部件串在一起有沒有問題」

### 🔹 UI / E2E Test

**驗證對象**：使用者在瀏覽器畫面上的完整操作流程。

- 登入、搜尋、加入購物車、結帳
- 訂單成立、錯誤訊息正確顯示
- 最接近使用者視角
- **不適合** 拿來取代所有測試，維護成本高

## 折扣功能的具體分工範例

以電商平台的折扣功能為例：

| 測試層 | 驗證什麼 | 不驗什麼 |
|--------|---------|---------|
| Unit Test | 折扣計算公式是否正確 | 畫面欄位、按鈕、API 串接 |
| Integration Test | 結帳流程是否正確套用折扣規則、依會員身分 + 購物車 + 優惠碼產生正確訂單摘要 | 畫面 UI 元素是否存在 |
| UI / E2E Test | 使用者能否在畫面完成購物流程、看到正確折扣明細與訂單金額、優惠碼欄位 + 套用按鈕 + 錯誤訊息是否存在且正確 | 底層計算邏輯細節 |

**反例**：只為了驗證一個折扣公式，就完整啟動瀏覽器 → 登入 → 加入購物車 → 前往結帳 → 測試成本過高，且失敗時難以定位是公式錯還是 UI 壞。

---

## 為什麼不能只靠 Unit Test？

Unit Test 驗不到：
- 畫面上是否真的有優惠碼欄位
- 套用按鈕是否存在且可點擊
- 折扣錯誤訊息是否正確顯示
- 訂單金額是否在正確的位置呈現

UI Test 才能回答「使用者看到的是什麼」。

---

## 分層設計原則

```
測試越靠近底層（Unit）→ 數量越多、速度越快、成本越低
測試越靠近頂層（E2E） → 數量越少、覆蓋最關鍵流程即可
```

合理分工：
- ✅ Unit Test：驗證單一邏輯（折扣計算、條件判斷、資料轉換）
- ✅ Integration Test：驗證元件整合（折扣邏輯 + 結帳 + 資料庫 + 會員身分）
- ✅ UI / E2E Test：驗證使用者畫面流程（欄位存在、按鈕可點、訊息正確、訂單成立）

## 和 AI 時代測試的關係

> 相關筆記：[[ai-testing-agile-quality]]、[[loop-engineering-testing]]

AI 自動生 code 的時代，測試分層更重要：
- AI 容易把所有驗證邏輯都寫進 E2E → 慢且難維護
- 要明確告訴 AI「這段邏輯用 unit test 驗，這段流程用 e2e 驗」
- Loop Engineering 的 stop condition 應該對應正確層次的測試，不是全部塞 E2E
