# AI Agent 上手第一課——一位醫師的 Claude Code 入門+治理心法

> 來源：派哥貼上的長文，作者是一位醫師（下週要在院內對復健科同仁講這個題目），系列分兩篇：「怎麼裝、怎麼跑起來」+「怎麼用得好」，本篇整理的是合併重點版
> 未附原文連結，內容為派哥直接貼上的全文

---

## 一、同一個人，只是工具長大了

作者自己的時間軸（三月到七月）：手動貼咒語盯著改 → 轉書/抽圖/OCR 磨成一鍵生產線 → 論文雷達/課程雷達自動推播 → 門診智慧病歷、額度守門、兩個 AI 互審。

> 「你現在就是我的三月。而且你比當年的我幸運，你一開始拿到的工具，就已經是好用的版本了。」

## 二、Claude Code 跟網頁版 ChatGPT/Claude 的根本差異

網頁版是「很會講話的人」，看不到你的檔案。Claude Code 是有手的：能讀資料夾裡的檔案、能寫/改檔案、能執行程式看結果、發現壞了自己回頭修——**在哪個資料夾打開它，它就能讀寫那裡**，不限於程式資料夾（逐字稿、衛教單資料夾都行）。

Agent 白話定義：**「你交代目標，他自己想步驟、自己動手、自己驗收。」**

## 三、安裝與內建安全機制

- 裝法：Windows PowerShell / Mac Terminal 貼一行指令，免 Node.js/Git/管理員權限，Pro 訂閱就夠
- 裝別人 GitHub 上的工具，正確做法是把 repo 網址丟給 Claude Code、叫它照 README 自己裝，**不要自己手動打路徑**（Mac/Windows 路徑寫法不同，agent 不會打錯，README 的英文它讀得懂）
- 內建安全機制：
  - 自動分辨動作輕重——讀檔/搜尋不打擾，會被問就代表要動到東西了，那正是該看一眼的時候。**「一直被問」是新手期的安全帶，先別急著關掉**
  - 改壞的檔案能用 `/rewind`（或按兩下 Esc）退回，像遊戲存檔
  - **邊界**：用 `rm`/`del` 指令刪的檔案不進資源回收桶、直接消失，`/rewind` 救不回
  - 對策三句話：練習期用複製品、會心痛的資料夾先備份、叫它「以後刪檔一律先移到資源回收桶，把這條記進規則檔」

## 四、真正決定成效的，是你怎麼設關卡（⚠️ 派哥點名要我特別學這段）

**最多人做錯的事**：把規則寫進 `CLAUDE.md` 抱怨它不聽話——但官方文件講得很白：「Claude treats them as context, not enforced configuration」，`CLAUDE.md` 的定位是參考資料，不是強制設定。

**真正管得住的三層機制，強度差很多：**

| 層 | 強度 | 機制 | 放什麼規則 |
|----|------|------|-----------|
| **① hooks** | ⭐⭐⭐ | 動手那一刻由程式攔下來檢查；被擋下的動作不是「勸退」，是「做不到」，模型不會回頭想替代方案 | 「這件事絕對不准發生」 |
| **② permissions** | ⭐⭐ | allow/ask/deny 三種清單，硬性擋下特定操作 | 「這類事要先問我，或永遠不准」 |
| **③ CLAUDE.md** | ⭐ | 每次開工都讀的文字說明，但靠自律，對話一長就被稀釋 | 「我們通常這樣做」的慣例 |

**新手最容易犯的錯**：覺得「一直要按同意好煩」就整個關掉自動確認。**正確做法是反過來**：安全的事放進 allow 讓它自己做，危險的事放進 deny 讓它永遠不能做，剩下才需要人判斷——注意力該留給真正需要判斷的那幾次。

**進階（自動模式）代價**：開自動模式連 `rm` 刪除也會一起放行，指令刪的檔案救不回。正確順序是**先把 deny/ask 設好、再開自動**——例如把 `rm` 設成 ask，之後不管切到哪一檔，刪除都會回到人手上。全自動也不是裸奔，官方後面還有一道安全分類器擋明顯越界動作，但那是最後一道網，不該是唯一一道。

## 五、最重要的一條心法：不要過度否決，也不要過度信任

AI 現在的程度像「很厲害的研究生」：讀得快、記得多、真的做得出來，但不保證不犯錯，**而且犯錯的時候語氣跟做對的時候一模一樣**——這才是真正麻煩的地方。

兩種極端都做不好事：
- **過度否決**：什麼都不敢交出去，等於把它當一台比較快的打字機
- **過度信任**：跑完直接用、直接貼進報告，出事才發現數字是湊的

**正確的問題不是「要不要相信他」，而是：「這件事如果錯了，我怎麼會知道？」** 答案是「不會知道」就要建檢核站：
- 能寫成程式的就寫成程式（分數/統計/日期讓程式算，AI 是語言模型不是計算機，算錯了還會講得很有自信）
- 能查證的就真的去查（引用存不存在，打 API 問，別靠它背）
- **失敗要講出來**：一項檢查沒跑成功必須明講「這項沒跑」，不能安靜跳過——因為沒跑的檢查讀起來跟「檢查過沒問題」一模一樣
- 不可逆動作要有關卡（就是上面 hooks/permissions/CLAUDE.md 三層在做的事）

> 「你不是在監工，你是在設計流程。人也會犯錯，醫院靠的不是『大家都很小心』，是 check list、雙人核對、機器警報。對 AI 也一樣。」

## 六、舊的 prompt 招式多半過時了

2023 年有用、現在多半白費力氣：❌ 開頭指派角色人格、❌ 落落長列 1234、❌ 手動整份資料貼進去（貼一次痛整場，讓它自己讀檔案就好）。

現在該做的是**給它一個能自己驗收的方法**：
- ❌「幫我整理這些逐字稿」
- ✅「幫我整理成 SOAP 格式。整理完自己檢查一遍：每份都要有這四段、每段都要有內容，沒有的標記出來給我」

有驗收標準的任務，它會做完、自己跑檢查、沒過自己修，一路迭代到通過；沒有的話只能做完就交，對錯要人自己看。

**零成本技巧**：複雜任務開頭加一句「先跟我講你打算怎麼做，不要動手」——看完計畫就會發現它誤會了什麼，這時候修比等它做完再修便宜太多。

## 七、一條紅線

雲端工具，打進去的東西會送給科技公司。涉及病人資訊先用假資料/去識別化資料練習，金鑰密碼一個都不要進去。不涉病人資訊的操作（格式轉換、寫衛教單、整理公開資料、自動化重複流程）放心用。**產出是素材不是結論**——最後掛名的臨床判斷還是要自己讀那篇論文。

## 對派哥的意義

**項目四是這篇對派哥最直接可用的一段**，因為它精準點出派哥自己 `~/.claude/CLAUDE.md` 現在的結構性弱點：那份全域規範裡列了一長串「天條」——API key 禁止明碼寫在腳本、gstack 禁止自動調用、Token/OAuth scope 單一 module 定義禁止複製貼上、大改版 push 前必須確認——**這些全部只活在 CLAUDE.md 這一層（強度⭐）**，靠的是我（Claude）每次開工自律遵守，對話一長就有被稀釋、漏看的風險。

按這篇的三層模型，真正該「做不到」而不是「靠自律記得」的規則，應該往上搬：
- **該是 hooks（⭐⭐⭐）的**：例如「API key 不能明碼寫進任何檔案」這種，理論上可以寫一個 pre-write hook 掃描要寫入的內容有沒有像 API key 的字串，擋下來而不是靠我自己記得檢查
- **該是 permissions（⭐⭐）的**：例如「大改版 push 前要確認」，可以把 `git push` 設成 ask（雖然目前 permission 系統本來就會問，但如果想要更嚴格區分「小修」vs「大改版」的判斷，靠的還是我自己的判斷，這塊目前沒有機制化）
- **留在 CLAUDE.md（⭐）就好的**：像「回答用繁體中文」「喜歡簡短直接」這種慣例性偏好，本來就不需要強制

這剛好呼應派哥已經在用的 [[claude-code-setup-plugin-2026-06|claude-code-setup 外掛]]（分析工作流建議 hooks/permissions/slash commands），可以找時間跑一次，順便把 CLAUDE.md 裡「絕對不准發生」等級的天條挑出來，問要不要升級成真正的 hook 或 permission deny 規則，而不是繼續全部放在自律層。

第五段「這件事如果錯了，我怎麼會知道」也是我在跑派哥的自動化任務（cc_processor、investment 排程、Notion sync）時該內化的檢核心法——之前 [[ai-testing-agile-quality]] 系列筆記講的「oracle problem」、失敗要講出來不能安靜跳過，這篇用醫療場景又講了一次同一件事，是跨領域的同一套判斷力。
