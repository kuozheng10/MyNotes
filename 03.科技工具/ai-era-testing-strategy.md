## 這是什麼

AI Coding 時代，寫測試的意義轉移：不再是「寫好寫滿」，而是「判斷哪裡值得測」。

---

## 核心論點

**AI 讓 Code 生成變便宜，但測試思維仍需人來判斷。**

- 寫測試 Code 本身 → AI 代勞，成本幾乎為零
- 維護測試 → 仍貴，需要人設計好
- 核心競爭力：找出「高 ROI 測試案例」的眼光

---

## 精準打擊原則

| 原則 | 做法 |
|------|------|
| 高 ROI 區域 | 核心業務邏輯，跳過 Setter/Getter |
| 守護關鍵路徑 | 登入、結帳等主功能永遠不能掛 |
| 風險驅動 | 壞掉損失大的地方 = 重兵防守 |
| 邊界條件 | AI 最容易在 Edge Case 翻車，人來把關 |

---

## 拒絕無效測試

一個「精準抓到潛在 Bug」的測試 > 一百個「只會綠燈」的虛假測試。

形式主義測試特徵：
- 只測實作細節（改 code 就壞）
- Assertion 毫無意義
- 由 AI 自動批量生成，無人審核

---

## 起步原則

> 每個 Story 完成後，只針對主功能寫 3–5 個自動化測試。**大於 0 永遠比 0 好。**

---

## 對派哥的啟示

- cc_processor / generate_report.py 等核心腳本：各加 3 個測試（正常/邊界/錯誤）比零測試強
- Sales report pivot 改壞過幾次 → 正是需要「關鍵路徑測試」保護的場景
- Claude Code 可以寫測試，但「定義測什麼」要派哥決定
