# AI 出來後，軟體開發哪些事變嚴重了？

> AI 是放大器，不是解方。DORA 2025 結論：放大高績效組織的強項，也放大失能組織的混亂。

---

## 核心命題

寫 code 變便宜了，但寫 code **前後左右的事沒變便宜**，瓶頸位移了。

---

## 五件變嚴重的事

### 1. 驗證變成瓶頸
AI 產出程式的速度 > 團隊理解它的速度（Thoughtworks）。  
瓶頸從「生成」轉到「理解與測試」。判斷一段程式對不對，最後還是落在人身上。

### 2. 上游失能被放大
需求一開始模糊 → AI 用快好幾倍的速度，照著模糊理解產出方向錯但語法正確的程式。  
以前頂多拖慢一個人，現在擴散到整個系統。  
→ **SBE（Spec by Example）投報率比以前高太多**

### 3. 整合瓶頸被放大

| 指標 | 數字 | 來源 |
|------|------|------|
| 吞吐量 | -1.5% | DORA 2024 |
| 穩定性 | -7.2% | DORA 2024 |
| PR 大小 | +51% | Faros 遙測 |
| PR review 中位時間 | **+441%** | Faros 遙測 |
| PR 無人審查直接 merge | **31%** | Faros 遙測 |

DORA 連兩年發現「用 AI 跟交付績效下降相關」：change list 變大、難整合。  
Conway's Law 沒失效，反而更明顯。

### 4. Review 與認知負荷
看 AI 寫的 code 特別累，因為少了寫的人腦中的 context，量又同時暴增。31% PR 沒審就 merge。

### 5. 理解在流失

| 指標 | 2021 | 2024 |
|------|------|------|
| 重構相關變更比例 | 25% | < 10% |
| 重複程式區塊 | 基準 | **+8×** |

（GitClear 分析 2.1 億行）  
程式還在長，但懂「為什麼這樣寫」的人越來越少。

---

## 反直覺數據：個人不一定變快

**METR 隨機對照試驗**：資深開發者在自己熟悉的成熟專案上用 AI，完成時間反而 **+19%**，而且他們自己以為變快了。

→ 個人變快不是普遍真理，要看情境。成熟 codebase + 資深工程師 → 可能剛好相反。

---

## 組織調整方向（四個）

| 方向 | 說明 |
|------|------|
| **往 feature team** | AI 降低跨領域學習成本，但架構決策還是要人對人 |
| **驗證端到端在團隊裡** | 減少 handoff，每多一次交接就多一個失控點 |
| **batch 更小、整合更頻繁** | DORA 數據最支持的一條 |
| **重新定義稀缺能力** | 從「誰能寫」→「誰能定義對、誰能驗證對」；Three Amigos / SBE / harness 擁有權是正式工作 |

---

## 診斷問題

> 先誠實盤點：你團隊的整合頻率跟驗證能力，撐不撐得住現在這個產出速度？

撐不住的話，AI 只會讓你更快累積出一堆沒人敢動的程式。
